# MySQL 数据定义语言

数据定义语言（Data Definition Language，DDL）用于定义和管理数据库对象的结构。常见的 DDL 语句包括：

- `CREATE`：创建数据库、表等对象。
- `ALTER`：修改已有对象的结构。
- `DROP`：删除数据库、表等对象。
- `TRUNCATE`：清空表中的全部数据，并保留表结构。
- `RENAME`：重命名表。

## 1. 数据库操作

### 1.1 查看数据库

```sql
SHOW DATABASES;
```

### 1.2 创建数据库

```sql
CREATE DATABASE [IF NOT EXISTS] 数据库名 [DEFAULT CHARACTER SET 字符集] [COLLATE 排序规则];
```

其中：

- `IF NOT EXISTS`：数据库不存在时才执行，避免执行失败。
- `DEFAULT CHARACTER SET`：设置数据库的默认字符集，决定字符如何编码和存储。`DEFAULT` 可以省略，并且 `CHARACTER SET` 可以简写为 `CHARSET`。
- `COLLATE`：设置数据库的排序规则，决定字符串如何比较和排序，例如是否区分大小写。

示例：

```sql
CREATE DATABASE IF NOT EXISTS learn_mysql
  DEFAULT CHARACTER SET utf8mb4
  COLLATE utf8mb4_0900_ai_ci;
```

查看数据库的创建语句：

```sql
SHOW CREATE DATABASE 数据库名;
```

### 1.3 选择数据库

```sql
USE 数据库名;
```

查看当前正在使用的数据库：

```sql
SELECT DATABASE();
```

### 1.4 删除数据库

```sql
DROP DATABASE [IF EXISTS] 数据库名;
```

其中：

- `IF EXISTS`：数据库存在时才执行。

删除数据库会同时删除其中的表和数据。

## 2. 数据类型

MySQL 中常用的数据类型主要分为三类：数值类型、字符串类型、日期时间类型。

### 2.1 数值类型

整数类型：

| 类型 | 存储空间 | 有符号取值范围 | 无符号取值范围 | 常见用途 |
| ---- | -------- | -------------- | ------------------- | -------- |
| `TINYINT` | 1 字节 | -128 ～ 127 | 0 ～ 255 | 年龄、较小的计数值 |
| `SMALLINT` | 2 字节 | -32,768 ～ 32,767 | 0 ～ 65,535 | 较小的编号 |
| `MEDIUMINT` | 3 字节 | -8,388,608 ～ 8,388,607 | 0 ～ 16,777,215 | 中等规模的编号、计数值 |
| `INT` 或 `INTEGER` | 4 字节 | -2,147,483,648 ～ 2,147,483,647 | 0 ～ 4,294,967,295 | 常用编号、数量、计数值 |
| `BIGINT` | 8 字节 | -9,223,372,036,854,775,808 ～ 9,223,372,036,854,775,807 | 0 ～ 18,446,744,073,709,551,615 | 大数据量表的主键、较大的计数值 |

不需要保存负数时，可以使用 `UNSIGNED`。例如：

```sql
age TINYINT UNSIGNED
```

定点数和浮点数：

| 类型 | 存储空间 | 精度或范围 | 特点 | 常见用途 |
| ---- | -------- | ---------- | ---- | -------- |
| `DECIMAL(M, D)` | 取决于 `M` 和 `D` | `M` 最大为 65，`D` 最大为 30，且 `D` 不能大于 `M` | 精确小数；`M` 是总位数，`D` 是小数位数 | 金额、财务数据 |
| `FLOAT` | 4 字节 | 约 7 位有效数字，最大绝对值约 3.4 × 10<sup>38</sup> | 单精度浮点数，可能产生舍入误差 | 对精度要求不高的测量值 |
| `DOUBLE` | 8 字节 | 约 15 ～ 16 位有效数字，最大绝对值约 1.8 × 10<sup>308</sup> | 双精度浮点数，可能产生舍入误差 | 科学计算、较高精度的近似值 |

`DECIMAL(10, 2)` 最多保存 10 位数字，其中 2 位是小数，可保存的值范围是 `-99999999.99` ～ `99999999.99`。

金额应优先使用 `DECIMAL`，不要使用可能产生精度误差的 `FLOAT` 或 `DOUBLE`。

MySQL 8.0.17 起禁止给 `DECIMAL`、`FLOAT` 和 `DOUBLE` 添加 `UNSIGNED`，需要限制小数不能为负数时，应使用 `CHECK` 约束：

```sql
price DECIMAL(10, 2) NOT NULL CHECK (price >= 0)
```

位值和布尔值：

| 类型 | 存储空间或范围 | 描述 | 常见用途 |
| ---- | -------------- | ---- | -------- |
| `BIT(M)` | `M` 为 1 ～ 64，约占 `⌈M / 8⌉` 字节 | 保存指定位数的位值 | 位掩码、多个开关状态 |
| `BOOLEAN` 或 `BOOL` | 1 字节 | 通常用 `0` 表示假、`1` 表示真 | 是否启用、是否删除等状态 |

`BOOLEAN` 本身不会限制字段只能保存 `0` 和 `1`。需要严格限制时，可以添加 `CHECK` 约束：

```sql
is_active BOOLEAN NOT NULL DEFAULT TRUE
  CHECK (is_active IN (FALSE, TRUE))
```

### 2.2 字符串类型

字符串类型可以保存字符或原始字节。字符字符串受字符集和排序规则影响；二进制字符串按字节保存和比较，不进行字符编码转换。

字符字符串：

| 类型 | 声明长度或容量上限 | 长度单位 | 特点 | 常见用途 |
| ---- | ------------------ | -------- | ---- | -------- |
| `CHAR(M)` | `M` 为 0 ～ 255 | 字符 | 定长字符串；未指定 `M` 时默认为 1 | 国家代码、定长编号 |
| `VARCHAR(M)` | `M` 为 0 ～ 65,535，实际容量受单行大小限制 | 字符 | 变长字符串，并额外保存内容长度 | 姓名、标题、邮箱、地址 |
| `TINYTEXT` | 255 字节 | 字节 | 很短的变长文本 | 简短说明 |
| `TEXT` | 65,535 字节 | 字节 | 常用的变长文本 | 文章内容、备注 |
| `MEDIUMTEXT` | 16,777,215 字节 | 字节 | 中等长度的变长文本 | 较长文章、日志内容 |
| `LONGTEXT` | 4,294,967,295 字节 | 字节 | 超长的变长文本 | 大型文本内容 |

`CHAR(M)` 和 `VARCHAR(M)` 中的 `M` 表示字符数，而不是字节数。例如，`CHAR(18)` 最多保存 18 个字符。`CHAR` 适合长度固定或基本固定的数据，`VARCHAR` 适合长度变化较大的数据。

字符数据实际占用的字节数与字符集有关。使用 `utf8mb4` 时，一个字符最多占 4 字节，因此 `VARCHAR(M)` 的实际最大长度还会受到 65,535 字节的单行大小、其他字段以及存储开销的共同限制。

二进制字符串：

| 类型 | 声明长度或容量上限 | 长度单位 | 特点 | 常见用途 |
| ---- | ------------------ | -------- | ---- | -------- |
| `BINARY(M)` | `M` 为 0 ～ 255 | 字节 | 定长二进制字符串；未指定 `M` 时默认为 1 | 定长哈希值、定长字节数据 |
| `VARBINARY(M)` | `M` 为 0 ～ 65,535，实际容量受单行大小限制 | 字节 | 变长二进制字符串 | 变长哈希值、变长字节数据 |
| `TINYBLOB` | 255 字节 | 字节 | 很小的二进制对象 | 小型二进制内容 |
| `BLOB` | 65,535 字节 | 字节 | 常用的二进制对象 | 文件、图片等二进制内容 |
| `MEDIUMBLOB` | 16,777,215 字节 | 字节 | 中等大小的二进制对象 | 较大的文件内容 |
| `LONGBLOB` | 4,294,967,295 字节 | 字节 | 超大的二进制对象 | 大型文件内容 |

`BINARY(M)` 和 `VARBINARY(M)` 中的 `M` 表示字节数。

除非确实需要把文件内容存入数据库，否则通常只在表中保存文件路径或对象存储地址。

枚举和集合：

| 类型 | 成员数量上限 | 描述 | 常见用途 |
| ---- | ------------ | ---- | -------- |
| `ENUM('值1', '值2', ...)` | 65,535 个 | 从预先定义的成员中选择一个值 | 状态、固定类别 |
| `SET('值1', '值2', ...)` | 64 个 | 从预先定义的成员中选择零个或多个值 | 多选标签、权限组合 |

`ENUM` 和 `SET` 的括号中是候选成员，不是最大长度 `M`。如果候选值经常变化，通常使用关联表比修改字段定义更容易维护。

### 2.3 日期时间类型

| 类型 | 基础存储空间 | 格式 | 有效范围 | 时区转换 | 常见用途 |
| ---- | ------------ | -------- | -------- | -------- | -------- |
| `YEAR` | 1 字节 | `YYYY` | 1901 ～ 2155 | 否 | 毕业年份、出版年份 |
| `DATE` | 3 字节 | `YYYY-MM-DD` | `1000-01-01` ～ `9999-12-31` | 否 | 生日、入学日期 |
| `TIME` | 3 字节 | `[-]HHH:MM:SS[.ffffff]` | `-838:59:59` ～ `838:59:59` | 否 | 每日时间、持续时长、时间差 |
| `DATETIME` | 5 字节 | `YYYY-MM-DD HH:MM:SS[.ffffff]` | `1000-01-01 00:00:00` ～ `9999-12-31 23:59:59` | 否 | 预约时间、业务发生时间 |
| `TIMESTAMP` | 4 字节 | `YYYY-MM-DD HH:MM:SS[.ffffff]` | `1970-01-01 00:00:01 UTC` ～ `2038-01-19 03:14:07 UTC` | 是 | 创建时间、更新时间、统一时间点 |

表中的存储空间不包含小数秒部分。`TIME`、`DATETIME` 和 `TIMESTAMP` 可以通过 `fsp` 指定 0 ～ 6 位小数秒精度，默认值为 0。例如，`DATETIME(3)` 可以保存到毫秒。根据精度不同，小数秒还会额外占用 0 ～ 3 字节。

#### 2.3.1 `TIME` 的两种用途

`TIME` 既可以表示一天中的时间，也可以表示持续时长或时间差。例如，`36:30:00` 表示 36 小时 30 分钟，`-02:15:00` 表示负 2 小时 15 分钟。

如果字段只允许保存一天中的时间，可以使用 `CHECK` 将范围限制在 24 小时以内：

```sql
start_time TIME NOT NULL
  CHECK (start_time >= '00:00:00' AND start_time < '24:00:00')
```

#### 2.3.2 `DATETIME` 和 `TIMESTAMP` 的区别

`DATETIME` 按写入值原样保存，不根据会话时区进行转换，取值范围也更大。它适合保存日历上的本地日期时间，以及 1970 年以前或 2038 年以后的时间。

`TIMESTAMP` 内部基于 UTC 保存，写入和读取时会根据当前会话的 `time_zone` 进行转换，适合表示跨时区的统一时间点。但它不会保存原始时区名称或时区偏移，并且存在 2038 年上限。使用 `TIMESTAMP` 时，应用程序应统一管理连接的会话时区。

#### 2.3.3 类型选择建议

- 只需要年份：使用 `YEAR`。
- 只需要日期：使用 `DATE`。
- 表示每日时间、持续时长或时间差：使用 `TIME`。
- 表示不需要时区转换的日历时间，或超出 `TIMESTAMP` 范围的时间：使用 `DATETIME`。
- 表示需要按会话时区转换且不超过 2038 年的统一时间点：使用 `TIMESTAMP`。

## 3. 表操作

### 3.1 查看表

查看当前数据库中的表：

```sql
SHOW TABLES;
```

查看表结构：

```sql
DESCRIBE 表名;
```

`DESCRIBE` 可以简写为 `DESC`。

查看完整的建表语句：

```sql
SHOW CREATE TABLE 表名;
```

### 3.2 创建表

```sql
CREATE TABLE [IF NOT EXISTS] 表名 (
  字段名 数据类型 字段约束[COMMENT 字段注释],
  字段名 数据类型 字段约束[COMMENT 字段注释],
  ...
  表级约束
)[CHARSET=字符集] [COLLATE=排序规则] [COMMENT 表注释];
```

创建表时，可以指定默认字符集和排序规则。如果没有指定，表会继承数据库的默认设置。

注释需要使用引号包裹起来。

## 4. 约束

约束用于保证数据的完整性和一致性。

| 约束 | 作用 |
| ---- | ---- |
| `PRIMARY KEY` | 唯一标识一行数据，值不能重复或为 `NULL` |
| `NOT NULL` | 字段值不能为 `NULL` |
| `UNIQUE` | 字段值不能重复 |
| `DEFAULT` | 未提供字段值时使用默认值 |
| `CHECK` | 字段值必须满足指定条件 |
| `FOREIGN KEY` | 限制字段值必须引用另一张表中存在的值 |

### 4.1 主键

一张表只能有一个主键，但主键可以由多个字段组成。

单字段主键：

```sql
CREATE TABLE courses (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(100) NOT NULL
);
```

联合主键：

```sql
CREATE TABLE student_courses (
  student_id BIGINT UNSIGNED NOT NULL,
  course_id BIGINT UNSIGNED NOT NULL,
  selected_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (student_id, course_id)
);
```

联合主键可以保证同一个学生不会重复选择同一门课程。

### 4.2 外键

为选课表添加外键：

```sql
ALTER TABLE student_courses
  ADD CONSTRAINT fk_student_courses_student
    FOREIGN KEY (student_id) REFERENCES students (id)
    ON UPDATE CASCADE
    ON DELETE CASCADE,
  ADD CONSTRAINT fk_student_courses_course
    FOREIGN KEY (course_id) REFERENCES courses (id)
    ON UPDATE CASCADE
    ON DELETE RESTRICT;
```

常用的引用操作包括：

- `CASCADE`：父表记录更新或删除时，同步更新或删除子表记录。
- `RESTRICT`：存在子表引用时，拒绝更新或删除父表记录。
- `SET NULL`：父表记录更新或删除时，将子表外键设为 `NULL`；外键字段必须允许为 `NULL`。
- `NO ACTION`：在 MySQL InnoDB 中，效果与 `RESTRICT` 基本相同。

外键字段与被引用字段的数据类型应保持一致。例如，两边都应使用 `BIGINT UNSIGNED`，不能一边有 `UNSIGNED`、另一边没有。

## 5. 修改表结构

使用 `ALTER TABLE` 修改已有表。

### 5.1 添加字段

```sql
ALTER TABLE students
  ADD COLUMN email VARCHAR(100) NULL COMMENT '邮箱' AFTER name;
```

- `FIRST`：把新字段放到最前面。
- `AFTER name`：把新字段放到 `name` 字段后面。
- 如果不指定位置，新字段默认添加到最后。

### 5.2 修改字段类型或约束

```sql
ALTER TABLE students
  MODIFY COLUMN email VARCHAR(150) NOT NULL COMMENT '邮箱';
```

使用 `MODIFY COLUMN` 时，需要完整写出希望保留的数据类型和字段约束，否则原有属性可能丢失。

如果表中已有数据不符合新的类型或约束，修改操作会失败。例如，在存在 `NULL` 值时，不能直接把字段改为 `NOT NULL`。

### 5.3 重命名字段

MySQL 8.0 及以上版本可以使用：

```sql
ALTER TABLE students
  RENAME COLUMN email TO contact_email;
```

也可以使用 `CHANGE COLUMN` 同时重命名并重新定义字段：

```sql
ALTER TABLE students
  CHANGE COLUMN contact_email email VARCHAR(150) NOT NULL COMMENT '邮箱';
```

`CHANGE COLUMN` 后必须重新写出字段的数据类型和约束。

### 5.4 设置和删除默认值

```sql
ALTER TABLE students
  ALTER COLUMN score SET DEFAULT 60.00;
```

删除默认值：

```sql
ALTER TABLE students
  ALTER COLUMN score DROP DEFAULT;
```

### 5.5 添加和删除约束

添加唯一约束：

```sql
ALTER TABLE students
  ADD CONSTRAINT uk_students_email UNIQUE (email);
```

删除唯一约束时，需要使用对应的索引名：

```sql
ALTER TABLE students
  DROP INDEX uk_students_email;
```

添加检查约束：

```sql
ALTER TABLE students
  ADD CONSTRAINT chk_students_birth_date
    CHECK (birth_date IS NULL OR birth_date >= '1900-01-01');
```

MySQL 的 `CHECK` 表达式不能使用 `CURRENT_DATE` 等非确定性函数。因此，“出生日期不能晚于今天”这类随时间变化的规则，通常应由应用程序在写入数据时校验。

删除检查约束：

```sql
ALTER TABLE students
  DROP CHECK chk_students_birth_date;
```

删除外键约束：

```sql
ALTER TABLE student_courses
  DROP FOREIGN KEY fk_student_courses_course;
```

删除外键后，MySQL 自动创建的相关索引不一定会同时删除。如果不再需要该索引，可以确认索引名后单独执行 `DROP INDEX`。

### 5.6 删除字段

```sql
ALTER TABLE students
  DROP COLUMN email;
```

删除字段也会永久删除该字段中的数据。执行前应检查它是否被索引、约束、视图或应用程序使用。

## 6. 重命名表

可以使用 `RENAME TABLE`：

```sql
RENAME TABLE students TO student_info;
```

也可以使用 `ALTER TABLE`：

```sql
ALTER TABLE student_info RENAME TO students;
```

`RENAME TABLE` 还可以在一条语句中重命名多张表。

## 7. 清空表和删除表

### 7.1 使用 `TRUNCATE` 清空表

```sql
TRUNCATE TABLE students;
```

`TRUNCATE TABLE` 会删除表中的全部数据，但保留表结构，并通常将 `AUTO_INCREMENT` 计数器重置为初始值。它不能带 `WHERE` 条件。

如果其他表的外键正在引用该表，`TRUNCATE TABLE` 可能会执行失败。

### 7.2 使用 `DROP` 删除表

```sql
DROP TABLE students;
```

避免表不存在时执行失败：

```sql
DROP TABLE IF EXISTS student_courses;
```

一次删除多张表：

```sql
DROP TABLE IF EXISTS student_courses, courses, students;
```

存在外键关系时，通常应先删除子表，再删除父表。

### 7.3 `DELETE`、`TRUNCATE` 和 `DROP` 的区别

| 语句 | 删除数据 | 删除表结构 | 支持 `WHERE` | 重置自增值 |
| ---- | -------- | ---------- | ------------ | ---------- |
| `DELETE` | 是 | 否 | 是 | 通常不重置 |
| `TRUNCATE TABLE` | 全部删除 | 否 | 否 | 通常重置 |
| `DROP TABLE` | 全部删除 | 是 | 否 | 表已不存在 |

`DELETE` 属于 DML，`TRUNCATE TABLE` 和 `DROP TABLE` 属于 DDL。

## 8. DDL 的执行特点

MySQL 中大多数 DDL 语句会隐式提交当前事务，通常不能通过 `ROLLBACK` 撤销。MySQL 8 的原子 DDL 可以保证受支持的 DDL 操作整体成功或整体失败，但这不等于执行成功后还能回滚。

执行 DDL 前应注意：

- 确认当前连接的数据库，必要时先执行 `SELECT DATABASE();`。
- 修改或删除结构前，先使用 `SHOW CREATE TABLE` 保存原始定义。
- 删除数据库、表或字段前做好备份。
- 修改大表结构可能耗时较长，并可能影响并发读写。
- 先在开发或测试环境验证，再应用到生产环境。
- 生产环境应通过版本化的数据库迁移脚本管理结构变更。

## 9. 完整练习

以下示例创建课程表和选课表，并建立主键、唯一约束和外键关系：

```sql
CREATE DATABASE IF NOT EXISTS ddl_practice
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_0900_ai_ci;

USE ddl_practice;

CREATE TABLE students (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  student_no VARCHAR(20) NOT NULL,
  name VARCHAR(50) NOT NULL,
  created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT uk_students_student_no UNIQUE (student_no)
) ENGINE = InnoDB;

CREATE TABLE courses (
  id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  course_no VARCHAR(20) NOT NULL,
  name VARCHAR(100) NOT NULL,
  credit DECIMAL(3, 1) NOT NULL,
  CONSTRAINT uk_courses_course_no UNIQUE (course_no),
  CONSTRAINT chk_courses_credit CHECK (credit > 0)
) ENGINE = InnoDB;

CREATE TABLE student_courses (
  student_id BIGINT UNSIGNED NOT NULL,
  course_id BIGINT UNSIGNED NOT NULL,
  selected_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (student_id, course_id),
  CONSTRAINT fk_student_courses_student
    FOREIGN KEY (student_id) REFERENCES students (id)
    ON UPDATE CASCADE
    ON DELETE CASCADE,
  CONSTRAINT fk_student_courses_course
    FOREIGN KEY (course_id) REFERENCES courses (id)
    ON UPDATE CASCADE
    ON DELETE RESTRICT
) ENGINE = InnoDB;

SHOW TABLES;
SHOW CREATE TABLE student_courses;
```

练习完成后，可以按引用关系从子表到父表依次删除：

```sql
DROP TABLE IF EXISTS student_courses;
DROP TABLE IF EXISTS courses;
DROP TABLE IF EXISTS students;
```
