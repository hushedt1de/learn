# mattpocock/skills 使用指南

`mattpocock/skills` 是一组面向真实工程工作的 AI agent skills。它不是让 agent 接管整个开发流程，而是把常见工程纪律拆成小、可组合、可修改的技能。

## 它主要解决什么问题

### 1. Agent 没有真正理解需求

很多时候，问题不是 agent 不会写代码，而是它太早开始写代码。用户以为自己说清楚了，agent 也以为自己理解了，但最后实现出来才发现双方对目标、边界、优先级和例外情况的理解不一样。

这类问题用 `/grill-me` 或 `/grill-with-docs` 处理。它们会让 agent 先追问，把需求、约束、分支情况和验收标准问清楚。

### 2. Agent 不理解项目里的“黑话”

真实项目里经常有大量内部术语、业务名词和历史决策。新人需要时间理解，AI agent 也一样。如果没有共享语言，agent 往往会用很长的描述绕来绕去，而且容易命名不一致。

这类问题用 `/grill-with-docs` 和 `domain-modeling` 处理。它们会帮助维护项目的 `CONTEXT.md`，把项目中的领域词汇、核心概念和容易误解的规则记录下来。

### 3. 代码写出来但不可靠

AI 很容易快速写出看起来合理的代码，但如果没有反馈循环，就不知道它是否真的工作。

这类问题用 `/tdd` 和 `/diagnosing-bugs` 处理。前者强调 red-green-refactor，也就是先写失败测试，再写代码让测试通过，最后重构；后者强调复现、缩小范围、提出假设、插桩验证、修复和回归测试。

### 4. 项目逐渐变得难维护

AI 写代码速度越快，项目复杂度堆积得也越快。如果不持续关心模块边界、接口设计和代码组织，项目会很快变成难以修改的结构。

这类问题用 `/improve-codebase-architecture` 和 `codebase-design` 处理。它们会提醒你关注会影响哪些模块、是否有更清晰的边界、是否需要把设计决策记录下来。

## 几个关键概念

### Issue tracker

Issue tracker 是用来记录、分配、跟踪工作的系统。常见形式包括 GitHub Issues、Linear、Jira、GitLab Issues，也可以是本地 Markdown 文件。

一个 issue 可以是 bug、feature、task、improvement 或 question。Issue tracker 通常会记录标题、描述、状态、标签、负责人、评论和关联 PR。

在这组 skills 里，issue tracker 是 `/to-prd`、`/to-issues` 和 `/triage` 这些 skill 的工作承载地。

### `CONTEXT.md`

`CONTEXT.md` 是这组 skills 推荐维护的项目上下文文件。它不是框架标准文件，而是给 AI agent 和项目维护者看的“共享语言文档”。

它通常记录：

- 项目里的核心业务概念。
- 专有名词和缩写。
- 模块之间的关系。
- 容易误解的领域规则。
- 哪些词应该怎么用，哪些词不要混用。
- agent 后续进入项目时必须知道的背景。

### ADR

ADR 是 Architecture Decision Record，也就是架构决策记录。它用来记录某个重要技术或架构选择为什么这么做。

例如：

- 为什么选择某个数据库。
- 为什么不用某个框架。
- 为什么把某个模块拆出来。
- 为什么采用某种状态管理方式。

`/grill-with-docs` 在追问需求时，如果发现有重要决策值得长期保留，就可能建议把它写成 ADR。

## 最简单的选择方式

如果不确定该用哪个 skill，先用 `ask-matt`。

`ask-matt` 在 README 里被描述为一个 router，也就是“我现在该用哪个 skill”的入口。你可以先描述当前情况，让它根据你的目标推荐合适流程。

## 按场景选择

| 你的场景 | 适合的 skill |
| --- | --- |
| 我有个想法，但还没想清楚 | `/grill-me` |
| 我要做代码功能，而且希望顺便沉淀项目术语和决策 | `/grill-with-docs` |
| 我们已经聊完了，想整理成产品需求文档 | `/to-prd` |
| 我有 PRD/计划，想拆成可执行任务 | `/to-issues` |
| 我想先快速试几个方案，不想直接进正式代码 | `/prototype` |
| 我要修复杂 bug 或性能问题 | `/diagnosing-bugs` |
| 我要实现功能，并希望测试先行 | `/tdd` |
| 项目越来越乱，想找架构改进点 | `/improve-codebase-architecture` |

## `grill-me` 和 `grill-with-docs` 的区别

这两个 skill 都是在“动手之前先问清楚”。区别在于它们是否绑定到一个具体代码库，以及是否需要更新项目长期上下文。

`/grill-me` 是通用追问。它适合没有明确绑定代码库的想法、计划、产品方向、写作、学习或决策。

例如：

- 我想设计一个英语学习计划。
- 我想规划一个个人知识库系统。
- 我想想清楚一个 SaaS 产品的商业模式。
- 我想写一篇文章，但主题还很散。
- 我想决定这个功能到底值不值得做。

这些问题可能以后会变成代码，但当前重点是想清楚目标、约束、受众、方案和取舍。

`/grill-with-docs` 是面向代码库的追问。它适合你已经有一个项目，并且这次需求会影响它。

例如：

- 给当前项目加登录功能。
- 修改计费逻辑。
- 重构订单导出模块。
- 修复课程发布流程里的状态问题。
- 新增一个后台管理页面。

它除了追问需求，还会关心这次改动影响哪些模块、项目里已有术语是什么、是否需要更新 `CONTEXT.md`、有没有架构决策要写进 ADR。

一个简单判断方法：

> 这次讨论结束后，是否应该更新项目里的长期上下文文档？

如果答案是“是”，用 `/grill-with-docs`。如果只是先把想法聊清楚，用 `/grill-me`。

## 典型工程流程

README 里隐含的主流程大概是：

1. 先运行 `/setup-matt-pocock-skills` 进行项目初始化

   配好 issue tracker、triage 标签、文档存放位置。
   通常每个代码仓库只需运行一次。

2. 有新需求时运行 `/grill-me` 或者 `/grill-with-docs`

   让 agent 先追问，不急着写代码。

3. 需求清楚后运行 `/to-prd`

   把当前讨论整理成 PRD。

4. 再用 `/to-issues`

   把 PRD 拆成独立、可领取、垂直切片的 issue。

5. 实现时根据情况用 `/tdd` 或 `/diagnosing-bugs`

   新功能偏 `/tdd`，疑难 bug 偏 `/diagnosing-bugs`。

6. 隔一段时间运行 `/improve-codebase-architecture`

   检查代码库有没有模块边界、复杂度、设计退化的问题。

## 初学者建议

刚开始用时，不需要一次性理解所有 skill。先记住这几个：

- `/grill-with-docs`：开始一个重要代码改动前用。
- `/to-prd`：把讨论整理成需求文档。
- `/to-issues`：把需求拆任务。
- `/tdd`：实现新功能时用测试驱动推进。
- `/diagnosing-bugs`：修复杂 bug 时用结构化诊断流程。

如果仍然不确定，就先用 `ask-matt`。它就是专门为“我该选哪个 skill”这个问题准备的入口。
