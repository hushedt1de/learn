# Redis 持久化

## 1. 什么是持久化

Redis 的数据主要保存在内存中。内存读写速度很快，但进程退出、服务器重启、机器宕机后，内存中的数据就会丢失。

持久化就是把 Redis 内存中的数据保存到磁盘文件中。Redis 重启后，可以从磁盘文件重新加载数据，让服务恢复到之前的状态。

简单理解：

```text
Redis 运行时
    |
数据保存在内存中
    |
通过持久化写入磁盘
    |
Redis 重启后从磁盘恢复数据
```

Redis 的持久化不是为了提高访问速度，而是为了提高数据恢复能力。平时读写仍然主要发生在内存中，磁盘文件主要用于故障恢复、重启恢复、备份迁移等场景。

## 2. 为什么需要持久化

### 2.1 防止 Redis 重启后数据全部丢失

如果 Redis 只作为纯缓存使用，数据丢失后可以从 MySQL 等后端数据库重新加载，影响可能只是短时间缓存命中率下降。

但很多业务会把一些重要状态也放到 Redis 中，例如：

- 登录 Session。
- 验证码和临时令牌。
- 限流计数。
- 排行榜数据。
- 秒杀库存预扣状态。
- Stream 消息。
- 分布式任务进度。

如果这些数据在 Redis 重启后全部丢失，可能会导致用户重新登录、任务重复执行、消息丢失、计数异常，甚至影响业务正确性。

### 2.2 缩短故障恢复时间

没有持久化时，Redis 重启后是一个空实例。应用需要重新从数据库加载热点数据，缓存预热期间数据库压力会明显升高。

启用持久化后，Redis 可以从本地文件恢复数据。虽然恢复出来的数据可能不是故障前最后一刻的完整状态，但通常能保留大部分数据，减少冷启动压力。

### 2.3 支持备份和迁移

持久化文件可以用于备份、迁移和环境复制。例如把生产环境某个时间点的 RDB 文件复制到测试环境，快速得到一份近似真实的数据。

也可以定期保存持久化文件，用于误操作后的数据回滚分析。需要注意的是，Redis 持久化文件不是完整的审计日志，不能替代数据库备份和业务级审计。

### 2.4 提高 Redis 作为辅助存储时的可靠性

Redis 不一定只做缓存。有些场景会把 Redis 当作辅助存储使用，例如排行榜、计数器、延迟任务、Stream 消息队列等。

这类数据不一定都能轻易从数据库重新生成，因此需要持久化降低数据丢失风险。

## 3. 什么时候需要持久化

是否需要持久化，取决于 Redis 中的数据能不能丢、丢了以后能不能快速恢复。

### 3.1 建议开启持久化的场景

如果 Redis 保存了重要业务状态，通常应开启持久化。

常见场景包括：

1. **Redis 中有无法完全重建的数据**  
   例如排行榜分数、实时计数、任务进度、Stream 消息等。

2. **Redis 重启后不能接受长时间冷启动**  
   例如高并发系统中，缓存为空会让大量请求直接打到数据库，可能引发缓存雪崩。

3. **需要做数据备份或迁移**  
   例如定期备份 Redis 数据，或把数据迁移到另一台机器。

4. **Redis 作为缓存以外的存储组件**  
   例如把 Redis 用作轻量消息队列、会话中心、分布式协调组件时，通常不能把数据丢失视为完全无影响。

### 3.2 可以不启用持久化的场景

如果 Redis 只保存可丢弃、可快速重建的纯缓存数据，可以考虑不启用持久化。

例如：

- 商品详情缓存，真实数据保存在 MySQL。
- 用户信息缓存，真实数据保存在数据库。
- 接口响应缓存，数据可以重新计算。
- 临时热点缓存，丢失后最多影响短时间性能。

不启用持久化可以减少磁盘写入和恢复复杂度，但需要确保 Redis 为空时系统仍能承受回源压力。通常还要配合缓存预热、限流和降级策略。

### 3.3 不要把持久化等同于绝对不丢数据

Redis 持久化可以降低数据丢失风险，但不能保证任何情况下都不丢数据。

例如：

- RDB 是周期性快照，两次快照之间的数据可能丢失。
- AOF 根据刷盘策略不同，也可能丢失最近 1 秒或更多数据。
- 操作系统、磁盘、文件系统异常可能影响写入结果。
- 主从复制发生故障切换时，也可能丢失尚未复制到从节点的数据。

如果业务要求强事务、强一致、严格不丢数据，通常应该把 MySQL、PostgreSQL 等数据库作为主存储，Redis 作为缓存或辅助组件。

## 4. Redis 持久化方案有哪些

Redis 主要提供两种持久化方案：

- **RDB（Redis Database）**：在某个时间点生成数据快照。
- **AOF（Append Only File）**：把写命令追加到日志文件中。

此外，Redis 还支持：

- **RDB + AOF 混合持久化**：AOF 重写时使用 RDB 格式保存基础数据，再追加增量命令。
- **关闭持久化**：只把 Redis 当作纯内存缓存。

不同方案关注点不同：

| 方案 | 核心思路 | 恢复速度 | 数据丢失风险 | 文件体积 | 适合场景 |
| --- | --- | --- | --- | --- | --- |
| RDB | 定期生成内存快照 | 快 | 可能丢失最近一段时间数据 | 较小 | 备份、迁移、可接受少量丢失 |
| AOF | 记录每条写命令 | 相对慢 | 取决于刷盘策略，通常更小 | 较大 | 更重视数据完整性 |
| 混合持久化 | RDB 快照加 AOF 增量 | 较快 | 接近 AOF | 适中 | 生产环境常用折中方案 |
| 不持久化 | 不写磁盘 | 无需恢复 | 重启后全部丢失 | 无 | 纯缓存、可快速重建 |

## 5. 方案一：RDB 快照持久化

### 5.1 RDB 的基本思路

RDB 会在某个时间点把 Redis 内存中的数据生成一份快照文件。这个文件通常叫 `dump.rdb`。

可以把 RDB 理解成给 Redis 数据拍了一张照片：

```text
Redis 内存数据
    |
生成快照
    |
dump.rdb
```

Redis 重启时，如果发现 RDB 文件，就会读取该文件并恢复数据。

### 5.2 RDB 如何触发

RDB 可以通过自动触发或手动触发生成。

自动触发通常由配置控制。例如：

```conf
save 900 1
save 300 10
save 60 10000
```

含义是：

- 900 秒内至少有 1 次写入，生成 RDB。
- 300 秒内至少有 10 次写入，生成 RDB。
- 60 秒内至少有 10000 次写入，生成 RDB。

手动触发常见命令有两个：

```redis
SAVE
BGSAVE
```

`SAVE` 会在主线程中生成 RDB 文件，期间 Redis 不能处理其他命令，因此生产环境通常不使用。

`BGSAVE` 会 fork 出一个子进程，由子进程生成 RDB 文件，主进程继续处理请求。生产环境更常用 `BGSAVE`。

### 5.3 BGSAVE 的执行过程

`BGSAVE` 大致流程如下：

```text
客户端执行 BGSAVE
    |
Redis 主进程 fork 子进程
    |
子进程把内存数据写入临时 RDB 文件
    |
写入完成后替换旧的 RDB 文件
    |
主进程继续处理客户端请求
```

fork 子进程时，操作系统会使用写时复制（Copy On Write）。父子进程一开始共享相同的内存页，只有当主进程继续修改某些数据时，操作系统才会复制被修改的内存页。

这使得 `BGSAVE` 不需要完整复制一份 Redis 内存，但在写入频繁、数据量很大时，仍然可能带来额外内存消耗。

### 5.4 RDB 的优点

RDB 文件是紧凑的二进制快照，体积通常比 AOF 小，适合做备份和迁移。

Redis 启动时加载 RDB 文件通常比较快，因为它直接读取某个时间点的数据状态，不需要重新执行大量写命令。

RDB 的运行时开销相对较低。大部分时候 Redis 不需要持续写磁盘，只在生成快照时集中写入。

### 5.5 RDB 的缺点

RDB 最大的问题是可能丢失最近一段时间的数据。

例如配置为 5 分钟生成一次快照，如果 Redis 在下一次快照生成前宕机，那么上一次快照之后的写入都可能丢失。

另外，`BGSAVE` 需要 fork 子进程。数据量很大时，fork 可能导致短暂停顿；如果系统内存不足，`BGSAVE` 还可能失败。

RDB 更像是“定时存档”，不适合对数据完整性要求很高的场景单独使用。

### 5.6 RDB 适合什么场景

RDB 适合以下场景：

- 可以接受丢失最近一段时间的数据。
- 需要定期备份 Redis 数据。
- 需要较快的重启恢复速度。
- Redis 主要做缓存，但希望重启后不要完全变空。
- 需要把某个时间点的数据迁移到其他环境。

如果 Redis 中保存的是排行榜、计数器、消息等更重要的数据，只使用 RDB 通常不够，需要考虑 AOF 或混合持久化。

## 6. 方案二：AOF 追加日志持久化

### 6.1 AOF 的基本思路

AOF（Append Only File）会把 Redis 接收到的写命令追加到日志文件中。这个文件通常叫 `appendonly.aof`，或者在新版本中由多个 AOF 相关文件组成。

例如客户端执行：

```redis
SET name "redis"
INCR counter
LPUSH queue "job1"
```

AOF 会把这些写命令按 Redis 协议格式追加到文件里。Redis 重启时，会重新执行 AOF 中的写命令，从而恢复数据。

可以把 AOF 理解成记录操作过程：

```text
客户端写命令
    |
Redis 执行命令
    |
命令追加到 AOF 文件
    |
重启时重放命令恢复数据
```

### 6.2 AOF 如何开启

AOF 默认不一定开启，可以通过配置启用：

```conf
appendonly yes
```

常见相关配置：

```conf
appendfilename "appendonly.aof"
appendfsync everysec
```

`appendfsync` 决定 AOF 数据刷盘策略，它直接影响性能和数据丢失风险。

### 6.3 AOF 刷盘策略

Redis 写 AOF 时，通常先写入操作系统的文件缓冲区，再由操作系统刷到磁盘。`appendfsync` 用来控制 Redis 什么时候主动要求刷盘。

常见策略有三种：

| 策略 | 含义 | 性能 | 数据丢失风险 |
| --- | --- | --- | --- |
| `always` | 每次写命令都刷盘 | 最慢 | 最低 |
| `everysec` | 每秒刷盘一次 | 折中 | 通常最多丢失约 1 秒数据 |
| `no` | 交给操作系统决定 | 最快 | 较高 |

生产环境最常见的是：

```conf
appendfsync everysec
```

它在性能和可靠性之间取得平衡。即使 Redis 或服务器异常退出，通常也只会丢失最近约 1 秒的数据。

`always` 更可靠，但每条写命令都需要等待刷盘，性能开销很大，一般只在极少数对数据丢失非常敏感且写入量不大的场景使用。

`no` 性能最好，但刷盘完全由操作系统控制，故障时可能丢失更多数据。

### 6.4 AOF 重写

随着 Redis 不断执行写命令，AOF 文件会越来越大。但很多历史命令对最终结果已经没有必要。

例如：

```redis
SET count 1
SET count 2
SET count 3
```

恢复数据时只需要最终结果：

```redis
SET count 3
```

AOF 重写就是把当前内存中的最终数据状态重新整理成一份更小的 AOF 文件。

手动触发命令：

```redis
BGREWRITEAOF
```

自动重写可以通过配置控制：

```conf
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
```

含义是：当 AOF 文件体积相比上次重写后增长超过 100%，并且文件大小至少达到 64MB 时，触发自动重写。

### 6.5 AOF 重写的执行过程

AOF 重写不是简单压缩旧文件，而是根据 Redis 当前内存数据生成新的 AOF 内容。

大致流程如下：

```text
触发 AOF 重写
    |
Redis fork 子进程
    |
子进程根据当前内存数据生成新的 AOF 文件
    |
重写期间的新写命令继续追加到缓冲区
    |
新文件生成完成后合并增量写命令
    |
替换旧 AOF 文件
```

这样可以避免重写过程中丢失新写入的数据。

### 6.6 AOF 的优点

AOF 的数据完整性通常比 RDB 更好。使用 `appendfsync everysec` 时，故障后一般只会丢失最近约 1 秒的数据。

AOF 记录的是写命令，格式相对容易理解，也便于在某些情况下人工分析或修复。比如误执行了某条危险命令后，可以在停机状态下尝试编辑 AOF 文件去掉该命令，再启动 Redis。实际生产中这样操作要非常谨慎，并先备份原文件。

### 6.7 AOF 的缺点

AOF 文件通常比 RDB 文件大，因为它记录的是命令过程，而不是压缩后的时间点快照。

Redis 启动时需要重放 AOF 命令，数据量和命令数量很大时，恢复速度可能比 RDB 慢。

AOF 持续写磁盘，对磁盘 I/O 更敏感。如果磁盘性能差，可能影响 Redis 写入延迟。

AOF 重写同样需要 fork 子进程，也会带来额外 CPU、内存和磁盘开销。

### 6.8 AOF 适合什么场景

AOF 适合以下场景：

- Redis 中有比较重要的业务状态。
- 希望故障时尽量少丢数据。
- 可以接受一定磁盘写入开销。
- 对恢复速度要求不是唯一优先级。
- Redis 用于排行榜、计数器、Stream、任务队列等场景。

如果生产环境中 Redis 数据不能随意丢失，通常至少应考虑开启 AOF。

## 7. 方案三：RDB + AOF 混合持久化

### 7.1 为什么需要混合持久化

RDB 恢复快、文件小，但可能丢失较多数据。AOF 丢失数据少，但文件可能较大，恢复时需要重放命令，速度相对慢。

混合持久化就是为了结合两者优点：

- 用 RDB 格式保存某个时间点的完整数据，提升加载速度。
- 用 AOF 格式保存该时间点之后的增量写命令，减少数据丢失。

### 7.2 混合持久化的基本思路

开启混合持久化后，AOF 重写生成的新文件不再完全由 Redis 命令组成，而是分为两部分：

```text
AOF 文件
    |
前半部分：RDB 格式的完整快照
    |
后半部分：AOF 格式的增量写命令
```

Redis 重启加载时，先按 RDB 格式快速恢复基础数据，再重放后面的增量命令。

### 7.3 如何开启混合持久化

需要先开启 AOF：

```conf
appendonly yes
```

再开启 AOF 重写时使用 RDB 前缀：

```conf
aof-use-rdb-preamble yes
```

该配置开启后，混合持久化会在 AOF 重写时生效。也就是说，它不是替代 AOF，而是改变 AOF 重写后的文件格式。

### 7.4 混合持久化的优点

混合持久化恢复速度通常比纯 AOF 快，因为基础数据以 RDB 格式加载，不需要从头重放大量历史命令。

数据完整性又接近 AOF，因为最近的写入仍然通过 AOF 增量命令保存。

它适合大多数既希望减少数据丢失，又希望提升重启恢复速度的生产场景。

### 7.5 混合持久化的缺点

混合持久化文件的前半部分是 RDB 二进制格式，不像纯 AOF 那样容易人工阅读和编辑。

它仍然依赖 AOF 刷盘策略。如果配置为 `appendfsync everysec`，异常宕机时仍可能丢失最近约 1 秒的数据。

它也不能消除 fork、磁盘 I/O、AOF 重写等开销，只是在恢复速度和数据完整性之间做了更好的平衡。

### 7.6 混合持久化适合什么场景

混合持久化适合以下场景：

- 生产环境希望开启 AOF，但担心纯 AOF 恢复慢。
- Redis 数据量较大，重启恢复时间需要控制。
- 可以接受 AOF 的磁盘开销。
- 希望在可靠性和恢复速度之间取得平衡。

很多生产环境会选择 AOF 加混合持久化，并使用 `appendfsync everysec`。

## 8. 方案四：关闭持久化

### 8.1 基本思路

关闭持久化就是不让 Redis 把数据写入磁盘。Redis 完全作为内存缓存使用。

可以关闭 RDB 自动保存：

```conf
save ""
```

并关闭 AOF：

```conf
appendonly no
```

这样 Redis 重启后不会从本地持久化文件恢复数据。

### 8.2 优点

关闭持久化可以减少磁盘写入，配置和运行模型更简单。

如果 Redis 中的数据全部可以从数据库或其他服务重新生成，那么不持久化可以避免 RDB、AOF 重写、磁盘空间、恢复文件损坏等运维复杂度。

### 8.3 缺点

Redis 一旦重启或宕机，数据会全部丢失。

如果没有缓存预热和流量保护，Redis 重启后大量请求会回源到数据库，可能造成数据库压力暴增。

另外，某些看似“只是缓存”的数据，实际可能承担了业务状态作用。例如登录状态、验证码、限流计数、幂等 token 等。如果这些数据丢失，可能会影响用户体验或业务安全。

### 8.4 适合什么场景

关闭持久化只适合纯缓存场景，并且要满足：

- 数据丢失后可以从权威数据源恢复。
- Redis 为空时数据库能承受短时间回源压力。
- 已经有缓存预热、限流或降级措施。
- Redis 中没有重要业务状态。

如果不确定 Redis 数据能否丢失，通常不要轻易关闭持久化。

## 9. RDB 和 AOF 如何选择

### 9.1 只用 RDB

只用 RDB 的特点是恢复快、文件小、备份方便，但可能丢失最近一段时间的数据。

适合场景：

- Redis 主要作为缓存。
- 可以接受分钟级数据丢失。
- 更重视恢复速度和备份便利性。
- 写入量较大，不希望持续写 AOF。

示例选择：

```text
商品详情缓存、配置缓存、可重建的热点数据
```

### 9.2 只用 AOF

只用 AOF 的特点是数据丢失更少，但文件更大，恢复可能更慢。

适合场景：

- Redis 中有比较重要的数据。
- 希望故障后尽量少丢数据。
- 可以接受持续磁盘写入开销。

示例选择：

```text
排行榜、计数器、Stream 消息、任务状态
```

### 9.3 同时开启 RDB 和 AOF

Redis 同时开启 RDB 和 AOF 时，重启恢复通常优先使用 AOF，因为 AOF 一般包含更完整的数据。

这种方式可以同时获得：

- RDB 用于备份和快速快照。
- AOF 用于更细粒度的数据恢复。

如果开启了混合持久化，AOF 重写后的文件中也会包含 RDB 格式的基础数据。

### 9.4 推荐选择

常见选择如下：

| 场景 | 推荐方案 |
| --- | --- |
| 纯缓存，数据可重建 | 可以关闭持久化，或只开 RDB |
| 缓存数据较多，希望重启后快速恢复 | RDB |
| 有重要业务状态，希望少丢数据 | AOF，`appendfsync everysec` |
| 数据量较大，同时关注可靠性和恢复速度 | AOF + 混合持久化 |
| 需要定期备份和迁移 | RDB，或同时开启 RDB 和 AOF |

对多数生产系统来说，如果 Redis 不是纯缓存，比较稳妥的选择是：

```conf
appendonly yes
appendfsync everysec
aof-use-rdb-preamble yes
```

同时保留合理的 RDB 快照配置，用于备份和迁移。

## 10. 持久化相关风险和注意事项

### 10.1 磁盘空间

RDB 和 AOF 都会占用磁盘空间。AOF 重写期间还可能同时存在旧文件和新文件，短时间内需要更多磁盘空间。

如果磁盘写满，Redis 可能无法继续正常持久化，甚至影响写命令执行。生产环境需要监控磁盘使用率。

### 10.2 fork 开销

`BGSAVE` 和 `BGREWRITEAOF` 都需要 fork 子进程。Redis 数据量越大，fork 成本越高。

fork 期间 Redis 主进程可能出现短暂停顿。写入很频繁时，写时复制也会增加内存消耗。

### 10.3 磁盘 I/O 抖动

AOF 刷盘、RDB 生成、AOF 重写都会产生磁盘 I/O。如果磁盘性能不足，可能导致 Redis 延迟升高。

生产环境中，Redis 持久化文件最好放在性能稳定的磁盘上，并避免和其他高 I/O 服务争抢同一块磁盘。

### 10.4 持久化文件损坏

异常宕机、磁盘故障、文件系统问题都可能导致持久化文件损坏。

Redis 提供了一些检查和修复工具，例如：

```bash
redis-check-rdb dump.rdb
redis-check-aof --fix appendonly.aof
```

修复前应先备份原文件。修复工具可能会截断损坏部分，意味着部分数据会丢失。

### 10.5 持久化不等于备份

持久化文件通常保存在 Redis 所在机器上。如果机器磁盘损坏，本地持久化文件也可能丢失。

生产环境仍然需要独立备份策略，例如：

- 定期复制 RDB 文件到对象存储或备份服务器。
- 对关键数据保留多份历史备份。
- 定期演练从备份恢复。
- 重要业务数据仍以数据库为准。

## 11. 小结

Redis 持久化的核心目的，是让内存数据在重启或故障后具备恢复能力。

RDB 是快照方案，文件小、恢复快、适合备份，但可能丢失最近一段时间的数据。

AOF 是日志方案，记录写命令，数据丢失通常更少，但文件更大，恢复可能更慢，并且会带来持续磁盘写入。

混合持久化结合了 RDB 的恢复速度和 AOF 的数据完整性，是很多生产环境的常用选择。

如果 Redis 只做纯缓存，可以不启用持久化；如果 Redis 中保存了重要状态，就应该认真设计持久化、备份、监控和故障恢复策略。
