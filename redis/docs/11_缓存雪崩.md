# Redis 缓存雪崩

## 1. 什么是缓存雪崩

缓存雪崩是指大量缓存 key 在同一时间失效，或者 Redis 服务整体不可用，导致原本应该由缓存承接的请求突然全部打到数据库上。

典型流程如下：

```text
大量 key 同时过期
    |
请求集中进入
    |
查询 Redis：大量未命中
    |
查询 MySQL：请求暴增
    |
数据库压力过大，响应变慢或不可用
```

Redis 的作用是挡在数据库前面，承接大部分读请求。缓存雪崩发生时，这层保护突然消失，数据库会在短时间内承受远超平时的压力。严重时不仅数据库会被打垮，应用线程池、连接池、网关和下游服务也可能被拖慢，最终形成连锁故障。

## 2. 缓存雪崩产生的原因

缓存雪崩的核心原因是：**大量请求在同一时间绕过缓存，集中访问数据库**。

常见场景包括：

1. **大量 key 设置了相同过期时间**  
   例如系统启动时批量把商品、用户、配置数据写入 Redis，并统一设置 30 分钟过期。30 分钟后这些 key 同时失效，后续请求会集中查询数据库。

2. **热点数据同时过期**  
   首页商品、活动页、榜单、推荐列表等高频访问数据如果在同一时间过期，会让大量用户请求同时穿过缓存层。

3. **Redis 宕机或网络异常**  
   如果 Redis 实例不可用，应用无法读取缓存，所有原本命中 Redis 的请求都会转向数据库。

4. **缓存重建速度跟不上请求速度**  
   某些 key 过期后，需要查询数据库并重新写入缓存。如果请求量很大，而缓存重建没有并发控制，就可能出现大量线程同时查询数据库。

5. **缓存预热不足**  
   系统发布、Redis 重启或缓存被清空后，如果没有提前加载核心数据，用户请求会直接触发大量数据库查询。

## 3. 解决方案一：过期时间加随机值

### 3.1 基本思路

不要让大量 key 使用完全相同的过期时间，而是在基础过期时间上增加一个随机偏移，让 key 分散过期。

例如原本所有商品缓存都设置 30 分钟过期：

```redis
SET product:1001 "{...}" EX 1800
SET product:1002 "{...}" EX 1800
SET product:1003 "{...}" EX 1800
```

可以改成 30 分钟加 0 到 5 分钟随机值：

```text
product:1001 -> 1800 + 37 秒
product:1002 -> 1800 + 191 秒
product:1003 -> 1800 + 263 秒
```

这样即使这些 key 在同一批次写入 Redis，也不会在同一秒集中失效。

### 3.2 优点

实现简单，对业务逻辑改动很小。它能有效避免“批量写入、批量过期”的问题，是防止缓存雪崩最常见的基础手段。

对于商品详情、用户信息、文章详情等大量独立 key，随机过期时间通常非常有效。

### 3.3 缺点

随机过期时间只能缓解大量 key 同时失效，不能解决 Redis 整体不可用的问题。

另外，如果某个单独的热点 key 访问量特别大，即使只有它一个 key 过期，也可能造成大量请求同时查询数据库。这种情况更接近缓存击穿，需要配合互斥锁、逻辑过期等方案。

### 3.4 使用建议

随机值不宜过大，也不宜过小。过小无法有效打散过期时间，过大可能让缓存数据比预期保留更久。

常见做法：

- 基础过期时间 5 分钟，可以增加 0 到 60 秒随机值。
- 基础过期时间 30 分钟，可以增加 0 到 5 分钟随机值。
- 基础过期时间 24 小时，可以增加 0 到 1 小时随机值。

具体范围要结合业务对数据新鲜度的要求决定。

## 4. 解决方案二：缓存预热

### 4.1 基本思路

缓存预热是指在用户请求到来之前，提前把核心数据加载到 Redis 中。这样系统启动、发布、Redis 重启或缓存清空后，不会让第一批用户请求直接打到数据库。

典型流程如下：

```text
系统启动或发布完成
    |
加载核心商品、配置、榜单等数据
    |
写入 Redis，并设置合理过期时间
    |
开放流量或逐步放量
```

例如电商首页、活动页、热门商品详情、系统配置等数据，都适合提前写入缓存。

### 4.2 优点

缓存预热可以减少冷启动阶段的数据库压力，让系统在刚上线或 Redis 刚恢复时更平稳。

对于访问量集中、数据范围明确的业务，例如首页数据、运营配置、热门榜单，预热效果很明显。

### 4.3 缺点

缓存预热需要知道哪些数据值得提前加载。如果预热范围太小，保护效果有限；如果预热范围太大，又会增加 Redis 内存占用和启动耗时。

预热数据还可能变旧。预热后如果数据库数据发生变化，需要有更新缓存或删除缓存的机制。

### 4.4 使用建议

缓存预热适合与发布流程、定时任务或后台管理操作结合。

常见做法：

1. 系统启动时加载必要配置和基础字典。
2. 发布活动前提前加载活动页、商品列表和库存展示数据。
3. Redis 故障恢复后，先预热核心数据，再逐步恢复全部流量。
4. 对预热 key 也设置随机过期时间，避免预热后再次集中失效。

## 5. 解决方案三：多级缓存

### 5.1 基本思路

多级缓存是指在 Redis 之外，再增加本地缓存、CDN、网关缓存等缓存层。即使 Redis 短时间不可用，应用也可以从其他缓存层读取部分数据，减少数据库压力。

常见结构如下：

```text
请求进入
    |
本地缓存
    |
Redis
    |
MySQL
```

对于静态资源、页面片段、接口响应等场景，也可以使用 CDN 或网关缓存：

```text
用户请求
    |
CDN / 网关缓存
    |
应用本地缓存
    |
Redis
    |
数据库
```

### 5.2 优点

多级缓存可以提升系统抗风险能力。Redis 出现问题时，本地缓存或网关缓存还能承接一部分读请求，不至于让数据库瞬间暴露在全部流量面前。

本地缓存访问速度很快，适合存放配置、字典、热点数据等读多写少的数据。

### 5.3 缺点

多级缓存会增加一致性复杂度。数据更新时，需要考虑 Redis、本地缓存、CDN 或网关缓存之间的同步和失效顺序。

本地缓存还会占用应用内存。多实例部署时，每个实例都有自己的本地缓存，可能出现不同实例缓存状态不一致的问题。

### 5.4 使用建议

多级缓存适合读多写少、短时间不一致可以接受的场景。

常见做法：

- 系统配置、地区字典、分类树等数据可以放入本地缓存。
- 页面内容、静态资源、公开接口可以考虑 CDN 或网关缓存。
- 本地缓存要设置容量上限和过期时间，避免占用过多内存。
- 数据更新后，可以通过消息队列或发布订阅通知各应用实例删除本地缓存。

## 6. 解决方案四：限流、熔断和降级

### 6.1 基本思路

当缓存大面积失效或 Redis 不可用时，系统需要限制进入数据库的请求量，并在必要时快速失败或返回降级结果。

常见手段包括：

- 限流：限制接口、用户、IP 或服务实例的请求频率。
- 熔断：当数据库错误率或响应时间超过阈值时，暂时停止继续请求数据库。
- 降级：返回兜底数据、默认值、静态页面或稍旧的数据。

例如：

```text
Redis 大量未命中
    |
检测到数据库压力升高
    |
限制查询流量
    |
部分请求返回降级结果
```

### 6.2 优点

限流、熔断和降级可以在缓存层失效时保护数据库和核心服务，避免故障继续扩大。

这类方案不依赖某一个 key 是否过期，而是从系统整体流量和资源消耗角度进行保护。

### 6.3 缺点

限流可能误伤正常用户。熔断和降级会牺牲一部分功能完整性或数据实时性。

如果没有提前设计降级结果，故障发生时临时处理往往很被动。例如商品详情可以展示“稍后再试”，但支付、下单等核心链路不能随意返回假成功。

### 6.4 使用建议

限流和熔断要结合监控指标使用，例如 Redis 命中率、数据库 QPS、数据库连接数、接口错误率和响应时间。

常见做法：

1. 对非核心接口设置更严格的限流。
2. 数据库压力升高时，优先降级推荐、排行榜、评论等非核心数据。
3. 对核心链路保留更多资源，例如登录、下单、支付状态查询。
4. 提前设计可接受的兜底数据，避免故障时返回不可控结果。

## 7. 解决方案五：提高 Redis 可用性

### 7.1 基本思路

如果缓存雪崩是由 Redis 整体不可用引起的，就需要从 Redis 架构上提高可用性，减少单点故障。

常见方案包括：

- 主从复制：主节点负责写，从节点负责读或作为备用。
- 哨兵模式：主节点故障后自动进行故障转移。
- Redis Cluster：数据分片存储，提升容量和整体可用性。
- 监控告警：及时发现 Redis 延迟升高、内存不足、连接异常等问题。

### 7.2 优点

高可用架构可以降低 Redis 单点故障带来的风险。即使某个节点出现问题，也能通过故障转移或分片能力减少影响范围。

对于依赖 Redis 承接核心读流量的系统，高可用是基础要求。

### 7.3 缺点

Redis 高可用会增加部署、运维和排障复杂度。主从切换、网络分区、数据同步延迟、集群扩容都需要额外关注。

高可用也不能替代业务侧保护。即使 Redis 架构稳定，仍然可能因为大量 key 同时过期而发生雪崩。

### 7.4 使用建议

线上环境不应只部署单个 Redis 实例承接核心业务缓存。至少需要有主从、哨兵或托管云 Redis 的高可用能力。

同时要监控关键指标：

- Redis 可用性和连接数。
- Redis 内存使用率。
- Redis 命中率。
- 慢查询和命令耗时。
- 主从复制延迟。

## 8. 方案对比

| 方案 | 主要作用 | 优点 | 缺点 | 适合场景 |
| --- | --- | --- | --- | --- |
| 过期时间加随机值 | 打散 key 过期时间 | 简单直接，成本低 | 不能解决 Redis 整体不可用 | 大量 key 批量写入、批量过期 |
| 缓存预热 | 提前加载核心数据 | 缓解冷启动压力 | 需要维护预热范围和更新机制 | 首页、活动页、热门数据 |
| 多级缓存 | 增加缓存兜底层 | 提升抗风险能力 | 一致性和内存管理更复杂 | 读多写少、可接受短暂不一致 |
| 限流、熔断和降级 | 控制异常流量 | 能保护数据库和核心服务 | 可能牺牲体验或实时性 | 缓存大面积失效、突发流量 |
| 提高 Redis 可用性 | 减少缓存层故障 | 降低单点故障风险 | 运维复杂度更高 | 核心业务缓存、线上环境 |

## 9. 实际开发中的组合方案

实际项目中通常不会只依赖一种方案，而是多种方案组合：

```text
请求进入
    |
CDN / 网关缓存
    |
应用本地缓存
    |
Redis
    |
未命中时限流和并发控制
    |
查询数据库
    |
重建缓存，设置随机过期时间
```

一种常见组合是：

1. 写入缓存时给过期时间增加随机值，避免大批 key 同时失效。
2. 系统启动、发布和 Redis 恢复后，对核心数据进行缓存预热。
3. 对配置、字典、热点数据增加本地缓存或网关缓存。
4. Redis 和数据库都接入监控，发现命中率下降或数据库压力升高时及时告警。
5. 为核心接口配置限流、熔断和降级策略，避免故障扩散。
6. Redis 采用主从、哨兵、Cluster 或云 Redis 高可用方案，减少整体不可用风险。

这样可以同时处理几类问题：

- 大量 key 同时过期：由随机过期时间打散。
- 系统冷启动：由缓存预热缓解。
- Redis 短暂不可用：由多级缓存和高可用架构降低影响。
- 数据库压力升高：由限流、熔断和降级保护。

## 10. 总结

缓存雪崩的本质是缓存层在短时间内失去承接能力，导致大量请求集中进入数据库。

解决缓存雪崩时，应优先做到：

1. 不要让大量 key 使用完全相同的过期时间。
2. 核心数据在系统启动、发布或 Redis 恢复后要提前预热。
3. 对关键读场景使用多级缓存，提高缓存层韧性。
4. 当数据库压力升高时，要有限流、熔断和降级手段。
5. 线上 Redis 要具备高可用能力，并配合监控告警。

在简单业务中，过期时间加随机值和缓存预热通常已经能解决大部分问题。在高并发、大流量、强依赖缓存的业务中，还需要结合多级缓存、限流降级和 Redis 高可用架构一起使用。
