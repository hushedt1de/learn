# Redis 分布式锁

## 1. 什么是分布式锁

分布式锁是用来协调多个进程、多个服务实例或多台机器访问同一份共享资源的一种机制。

在单机程序中，如果多个线程要同时修改同一个变量，可以使用本地锁，例如 Java 中的 `synchronized`、`ReentrantLock`。但在分布式系统中，请求可能被负载均衡到不同机器上，本地锁只能约束当前进程，无法约束其他机器上的进程。

例如有 3 个应用实例同时处理同一个订单：

```text
应用实例 A
应用实例 B  ---->  同一个订单 order:1001
应用实例 C
```

如果只使用本地锁，每个实例都会认为自己拿到了锁，因为它们的锁对象不在同一个进程内。这样仍然可能出现重复扣库存、重复支付回调、重复执行任务等问题。

分布式锁的核心目标是：**在多个分布式节点之间，保证同一时间只有一个节点可以执行某段关键逻辑**。

使用 Redis 实现分布式锁时，通常会把某个 key 当作锁：

```text
SET lock:order:1001 request-id NX EX 30
```

含义是：

- `lock:order:1001` 表示锁的名称。
- `request-id` 表示当前锁的持有者标识。
- `NX` 表示只有 key 不存在时才设置成功。
- `EX 30` 表示锁 30 秒后自动过期，避免持有锁的进程异常退出后锁永远不释放。

如果设置成功，说明当前客户端获得锁；如果设置失败，说明锁已经被其他客户端持有。

## 2. 什么情况下需要使用分布式锁

是否需要分布式锁，关键看两个条件：

1. 是否有多个节点可能同时执行同一段逻辑。
2. 这段逻辑是否会修改共享资源，或者要求同一时间只能执行一次。

如果只是普通查询，或者操作本身天然幂等，通常不需要分布式锁。分布式锁会增加系统复杂度，也会带来等待、超时和失败处理成本，不应该为了“看起来更安全”而滥用。

### 2.1 防止重复扣减库存

秒杀、下单、抢券等场景中，多个请求可能同时操作同一个商品库存。

```text
用户 A 下单商品 1001
用户 B 下单商品 1001
用户 C 下单商品 1001
```

如果库存判断和库存扣减不是一个严格受控的原子过程，就可能出现超卖。可以对商品维度加锁，例如 `lock:stock:1001`，同一时间只允许一个请求执行库存检查和扣减逻辑。

不过，库存扣减也可以通过数据库行锁、乐观锁、Redis Lua 脚本等方式实现。分布式锁不是唯一方案，具体要看数据最终落在哪里、并发量多大、业务是否允许失败重试。

### 2.2 防止重复处理订单

支付回调、退款回调、物流回调等接口经常会重复通知。多个相同通知如果同时到达不同应用实例，就可能重复修改订单状态。

常见做法是按订单号加锁：

```text
lock:order:1001
```

拿到锁的实例检查订单状态并执行处理；其他实例获取锁失败后，可以稍后重试，或者直接返回处理中。

即使使用了分布式锁，订单状态更新仍然应该设计成幂等。例如已经支付成功的订单再次收到支付成功回调时，不能再次增加余额或重复发货。

### 2.3 防止定时任务重复执行

在集群部署中，每个应用实例都可能启动同一个定时任务。如果没有协调机制，任务可能被多台机器同时执行。

例如每天凌晨生成报表：

```text
实例 A：执行报表任务
实例 B：执行报表任务
实例 C：执行报表任务
```

可以在任务开始前尝试获取 `lock:job:daily-report`。只有获得锁的实例执行任务，其他实例跳过。

如果任务执行时间较长，需要合理设置锁过期时间，或者使用续期机制，避免任务没执行完锁就过期，导致其他实例又开始执行同一个任务。

### 2.4 防止缓存击穿时重复重建缓存

热点 key 过期后，大量请求可能同时发现缓存未命中，然后一起查询数据库并重建缓存。

```text
热点缓存过期
    |
大量请求同时未命中 Redis
    |
大量请求同时查询 MySQL
```

这时可以对热点 key 加互斥锁：

```text
lock:cache:product:1001
```

只有拿到锁的请求查询数据库并写回缓存，其他请求短暂等待后重新读取缓存，或者返回旧值。这样可以防止大量请求同时压到数据库。

### 2.5 防止共享资源并发修改

只要多个服务实例会并发修改同一份共享资源，就可能需要分布式锁。例如：

- 生成连续编号。
- 修改账户余额。
- 抢占某个任务。
- 更新同一份配置。
- 同步外部系统中的同一条记录。

但要注意，很多共享资源本身已经有并发控制能力。例如数据库可以使用唯一索引、事务、行锁和乐观锁；消息队列可以通过消费组控制分发；Redis 自身可以通过原子命令和 Lua 脚本完成一些操作。

分布式锁适合保护一段跨多个操作的业务流程，而不是替代所有并发控制手段。

## 3. 怎么实现分布式锁

Redis 实现分布式锁的基本思路是：

1. 加锁时使用原子命令创建一个带过期时间的 key。
2. 释放锁时只能释放自己持有的锁。
3. 业务执行时间可能超过锁过期时间时，需要续期或重新评估方案。
4. 获取锁失败时，需要有明确的等待、重试或失败策略。

### 3.1 加锁：使用 SET NX EX

推荐使用 `SET key value NX EX seconds` 一条命令完成加锁和设置过期时间。

```redis
SET lock:order:1001 9f1c8c7a-5e40-4f64-a4e4-1d5f2a7c97f1 NX EX 30
```

如果返回 `OK`，表示加锁成功。如果返回空，表示锁已经存在，加锁失败。

不要使用下面这种两步方式：

```redis
SETNX lock:order:1001 request-id
EXPIRE lock:order:1001 30
```

因为 `SETNX` 成功后，如果应用在执行 `EXPIRE` 之前宕机，锁就没有过期时间，可能变成死锁。

`SET key value NX EX seconds` 是单条原子命令，可以避免这个问题。

### 3.2 锁的 value 要使用唯一标识

锁的 value 不应该随便写成固定值，例如 `1` 或 `locked`。它应该是当前请求或当前线程的唯一标识，例如 UUID。

```text
lock:order:1001 -> request-id-abc
```

这样释放锁时可以判断：

- 如果 Redis 中的 value 等于自己的 request id，说明锁仍然属于自己，可以释放。
- 如果 Redis 中的 value 不等于自己的 request id，说明锁已经过期并被别人重新获得，不能释放。

这个判断非常重要。否则可能出现“误删别人锁”的问题。

典型问题如下：

```text
客户端 A 加锁成功，锁 10 秒过期
    |
客户端 A 执行业务超过 10 秒，锁自动过期
    |
客户端 B 加锁成功
    |
客户端 A 执行完成，直接 DEL lock
    |
客户端 B 的锁被误删
```

使用唯一标识后，客户端 A 释放锁前会发现锁的 value 已经不是自己的 request id，于是不会删除。

### 3.3 解锁：使用 Lua 脚本保证原子性

释放锁不能简单执行：

```redis
GET lock:order:1001
DEL lock:order:1001
```

因为 `GET` 和 `DEL` 是两条命令，中间可能发生锁过期和重新加锁，仍然可能误删别人持有的锁。

正确做法是使用 Lua 脚本，把“判断 value 是否相等”和“删除 key”放在 Redis 中原子执行：

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

执行时传入锁 key 和当前请求的唯一标识：

```redis
EVAL "if redis.call('GET', KEYS[1]) == ARGV[1] then return redis.call('DEL', KEYS[1]) else return 0 end" 1 lock:order:1001 request-id-abc
```

这样可以保证只有锁持有者才能释放锁。

### 3.4 设置合理的过期时间

锁必须设置过期时间，避免客户端宕机、网络中断或进程异常退出后锁一直存在。

过期时间不能太短，也不能太长：

- 太短：业务还没执行完，锁就过期了，其他客户端可能进入临界区。
- 太长：持有锁的客户端异常退出后，其他客户端要等待很久才能继续。

通常可以根据业务最大执行时间设置。例如某段逻辑正常 200 毫秒完成，偶尔 1 秒，可以把锁过期时间设置为 5 到 10 秒。

如果业务执行时间不稳定，应该考虑：

1. 拆短临界区，只把真正需要互斥的部分放进锁内。
2. 使用锁续期机制。
3. 改用数据库事务、消息队列串行化、任务调度框架等更适合长任务的方案。

### 3.5 锁续期

如果业务执行时间可能超过锁过期时间，可以在持有锁期间定期续期。

例如锁初始过期时间为 30 秒，后台线程每 10 秒检查一次。如果锁仍然属于当前请求，就把过期时间重新设置为 30 秒。

续期也必须判断锁的 value，不能盲目执行 `EXPIRE`：

```lua
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("EXPIRE", KEYS[1], ARGV[2])
else
    return 0
end
```

锁续期可以减少业务执行时间过长导致的锁提前过期问题。但它也会增加实现复杂度，还要处理续期线程异常、应用暂停、Redis 网络抖动等情况。

如果是 Java 项目，可以使用 Redisson。Redisson 的看门狗机制会在业务未完成时自动续期，减少手写续期逻辑的风险。

### 3.6 获取锁失败后的处理

获取锁失败时，不应该无限死等。常见处理方式有三种：

1. **快速失败**  
   适合不允许重复提交的接口。例如用户重复点击提交订单，获取锁失败可以直接返回“处理中，请稍后再试”。

2. **有限重试**  
   适合短时间内可能释放锁的场景。可以重试几次，每次等待几十毫秒，并加入随机抖动，避免大量请求同时重试。

3. **排队处理**  
   如果业务必须处理每个请求，可以考虑使用消息队列，把请求串行化消费，而不是让大量线程竞争分布式锁。

示例流程：

```text
尝试加锁
    |
成功：执行业务，最后释放锁
    |
失败：等待一小段时间后重试
    |
超过最大重试次数：返回处理中或失败
```

### 3.7 Redis 单实例锁的局限

基于单个 Redis 实例的分布式锁实现简单、性能高，适合很多业务场景。但它不是强一致锁。

例如在 Redis 主从架构中，客户端 A 在主节点加锁成功，但锁还没复制到从节点时主节点宕机。此时从节点被提升为新主节点，客户端 B 可能又成功获得同一把锁。

```text
A 在 Redis 主节点加锁成功
    |
锁还没同步到从节点
    |
主节点宕机，从节点成为新主节点
    |
B 在新主节点加锁成功
```

这样同一时间可能出现两个客户端都认为自己拿到了锁。

如果业务不能接受这种风险，需要更强的协调组件，例如 ZooKeeper、etcd，或者使用数据库事务和唯一约束来保证正确性。

### 3.8 Redlock 简述

Redlock 是 Redis 官方提出的一种基于多个独立 Redis 节点的分布式锁算法。它的大致思路是：

1. 客户端依次向多个独立 Redis 节点尝试加锁。
2. 只有在多数节点加锁成功，并且总耗时没有超过锁有效期时，才认为加锁成功。
3. 解锁时向所有节点释放锁。

Redlock 的目标是降低单个 Redis 节点故障导致锁失效的风险。

不过，Redlock 的正确性在业界有不少讨论。对于一般业务，使用 `SET NX EX + 唯一标识 + Lua 解锁` 通常已经足够；对于资金、交易、库存强一致等高风险场景，不应只依赖 Redis 锁，而应该结合数据库事务、幂等设计、唯一约束、对账补偿等机制。

### 3.9 一个完整流程

一个相对完整的 Redis 分布式锁流程如下：

```text
生成 request id
    |
SET lock:key request-id NX EX 30
    |
加锁成功？
    |
是：执行业务逻辑
    |
finally 中使用 Lua 脚本释放锁
    |
否：返回失败、稍后重试或进入队列
```

伪代码示例：

```text
requestId = UUID()

locked = SET lockKey requestId NX EX 30

if not locked:
    return "处理中，请稍后再试"

try:
    执行业务逻辑
finally:
    if GET lockKey == requestId:
        DEL lockKey
```

注意：上面伪代码中的 `GET + DEL` 只是为了表达逻辑，真实实现必须使用 Lua 脚本保证原子性。

## 4. 使用分布式锁的注意事项

### 4.1 锁的粒度要合适

锁粒度太大，会降低并发能力。例如所有下单请求都抢同一把 `lock:order`，系统会变成串行处理。

更好的做法是按业务对象加锁：

- 商品库存：`lock:stock:{productId}`
- 订单处理：`lock:order:{orderId}`
- 用户操作：`lock:user:{userId}`
- 缓存重建：`lock:cache:{cacheKey}`

锁粒度越小，并发能力通常越好。但粒度太细也会增加管理复杂度，需要结合业务冲突范围决定。

### 4.2 锁内逻辑要尽量短

锁保护的是临界区。临界区越长，其他请求等待时间越长，也越容易遇到锁过期、续期失败和重试风暴。

锁内尽量只放必须互斥的操作，不要放无关的远程调用、复杂计算、大文件处理等耗时逻辑。

### 4.3 分布式锁不能替代幂等

分布式锁可以减少并发冲突，但不能保证请求只发生一次。

网络重试、消息重复投递、支付平台重复回调、用户重复点击，都可能让同一业务请求多次到达系统。即使有锁，也可能因为锁过期、服务重启、执行失败等原因发生重复处理。

因此关键业务仍然要做幂等控制。例如：

- 订单状态只能按合法状态流转。
- 支付流水号建立唯一索引。
- 消息消费记录去重。
- 库存扣减使用版本号或原子扣减。

### 4.4 分布式锁不是强一致事务

Redis 分布式锁通常适合做并发互斥，而不是提供严格事务语义。

如果业务要求“绝对不能重复执行”“绝对不能多扣钱”“绝对不能超卖”，需要把最终正确性放在数据层和业务模型上，例如数据库事务、唯一约束、乐观锁、状态机、对账和补偿。

分布式锁可以作为降低并发冲突的手段，但不应该成为唯一的正确性保障。

## 5. 小结

分布式锁用于解决分布式环境下多个节点同时访问共享资源的问题。Redis 实现分布式锁时，最基础也最常见的方式是：

```redis
SET lock:key request-id NX EX seconds
```

并使用 Lua 脚本按 value 校验后释放锁。

实际使用时要重点关注：

- 加锁和设置过期时间必须是原子操作。
- 锁的 value 必须是唯一标识。
- 解锁必须校验锁归属，避免误删别人锁。
- 锁过期时间要结合业务执行时间设置。
- 获取锁失败要有明确策略，不能无限等待。
- 关键业务仍然需要幂等、事务、唯一约束和补偿机制兜底。

分布式锁的作用是协调并发，不是替代业务正确性设计。
