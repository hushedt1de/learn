# Redis 缓存穿透

## 1. 什么是缓存穿透

缓存穿透是指客户端请求的数据既不在缓存中，也不在数据库中。由于缓存无法命中，应用会继续查询数据库；数据库也查不到结果，因此通常不会把数据写回缓存。这样一来，后续相同请求仍然会绕过缓存，持续打到数据库。

典型流程如下：

```text
请求 user:999999
    |
查询 Redis：未命中
    |
查询 MySQL：不存在
    |
返回空结果，不写缓存
```

如果只是偶尔访问不存在的数据，影响通常不大。但如果大量请求集中查询不存在的 key，例如恶意攻击、爬虫扫描、参数被篡改或业务代码生成了错误 ID，就会让请求绕过 Redis，直接压到数据库上。严重时会导致数据库连接被打满、响应变慢，甚至影响正常用户访问。

## 2. 缓存穿透产生的原因

缓存穿透的核心原因是：**不存在的数据无法被缓存拦截**。

常见场景包括：

1. **请求非法 ID 或随机 key**  
   例如用户不断请求 `product:999999999`、`user:-1`、`order:abc` 等明显不存在的数据。

2. **业务数据本身不存在**  
   用户访问已删除的商品、已注销的账号、过期的活动页面。如果系统没有缓存空结果，每次访问都会查询数据库。

3. **接口缺少参数校验**  
   应用没有在进入缓存和数据库之前过滤非法参数，导致无效请求进入后端查询流程。

4. **恶意攻击或爬虫扫描**  
   攻击者构造大量不存在的 key，使缓存层失效，迫使数据库承担全部查询压力。

## 3. 解决方案一：缓存空值

### 3.1 基本思路

当 Redis 未命中，并且数据库也查不到数据时，不是直接返回，而是把一个空结果写入 Redis。例如使用空字符串、特殊 JSON、占位标记等表示“这个数据不存在”。

```text
请求 user:999999
    |
查询 Redis：未命中
    |
查询 MySQL：不存在
    |
写入 Redis：user:999999 -> 空值，设置较短过期时间
    |
返回空结果
```

下次再请求同一个 key 时，应用可以直接从 Redis 读到空值，不再访问数据库。

示例：

```redis
SET user:999999 "__NULL__" EX 60
```

应用读取到 `__NULL__` 时，就知道该用户不存在，可以直接返回空结果或 404。

### 3.2 优点

缓存空值实现简单，对已有缓存查询流程改动较小。它适合处理“重复访问同一个不存在 key”的场景，例如某个已删除商品被很多用户访问。

它还能避免短时间内同一个不存在数据反复查询数据库，把压力转移到 Redis。

### 3.3 缺点

缓存空值会额外占用 Redis 内存。如果攻击者持续生成大量随机 key，例如 `user:1`、`user:2`、`user:3` 一直变，就可能在 Redis 中产生大量空值缓存。

另外，空值缓存可能带来短暂的数据不一致。例如数据库中原本没有 `user:999999`，因此缓存了空值；随后这个用户被创建，但空值缓存还没过期，应用仍然会认为用户不存在。

### 3.4 使用建议

空值缓存通常要设置较短的过期时间，例如几十秒到几分钟，避免长期占用内存，也减少数据不一致时间。

对于不同业务可以设置不同过期时间：

- 用户、商品等普通数据：空值缓存可以设置 30 秒到 5 分钟。
- 活动页、配置项等变更较少的数据：可以适当延长。
- 安全敏感接口：不能只依赖空值缓存，还应配合参数校验和限流。

## 4. 解决方案二：布隆过滤器

### 4.1 基本思路

布隆过滤器（Bloom Filter）是一种用于判断“某个元素是否可能存在”的数据结构。它可以用很小的内存保存大量 key 的存在性信息。

在查询 Redis 之前，先用布隆过滤器判断请求的 key 是否可能存在：

```text
请求 product:1001
    |
布隆过滤器判断：可能存在
    |
查询 Redis
    |
缓存未命中再查询数据库
```

如果布隆过滤器判断 key 一定不存在，就直接返回，不再访问 Redis 和数据库：

```text
请求 product:999999999
    |
布隆过滤器判断：一定不存在
    |
直接返回空结果
```

### 4.2 为什么布隆过滤器能防止穿透

缓存穿透的问题在于不存在的数据进入了数据库查询流程。布隆过滤器放在缓存和数据库之前，相当于先做一次存在性拦截。

例如系统中只有这些商品 ID：

```text
1001, 1002, 1003
```

初始化时把这些 ID 加入布隆过滤器。后续请求 `product:999999` 时，布隆过滤器可以快速判断这个 ID 一定不存在，于是应用直接返回，不让请求继续打到数据库。

### 4.3 优点

布隆过滤器非常适合拦截大量随机不存在 key 的请求。它的内存占用比直接保存所有 key 小很多，判断速度也很快。

对于商品 ID、用户 ID、文章 ID 这类数据规模大但存在性边界明确的业务，布隆过滤器很常见。

### 4.4 缺点

布隆过滤器存在误判。它可能把不存在的数据判断为“可能存在”，但不会把已经加入的数据判断为“一定不存在”。

这意味着布隆过滤器不能完全阻止所有无效请求，但能过滤掉大部分明显不存在的请求。

另外，标准布隆过滤器不擅长删除元素。如果业务中数据频繁删除，需要考虑重建过滤器、使用计数布隆过滤器，或者接受短时间内删除数据仍被判断为“可能存在”。

### 4.5 使用建议

布隆过滤器适合数据量大、非法 key 多、查询入口集中的场景。

常见做法：

1. 系统启动或定时任务把数据库中的合法 ID 加入布隆过滤器。
2. 新增数据时，同时把新 ID 加入布隆过滤器。
3. 查询时先判断布隆过滤器，不存在则直接返回。
4. 对误判后进入数据库的请求，仍然可以配合空值缓存。

在 Redis 中可以使用 RedisBloom 模块提供的布隆过滤器能力，也可以在应用内存中维护过滤器。应用内存方案速度快，但多实例环境下需要考虑数据同步和重启加载。

## 5. 解决方案三：参数校验

### 5.1 基本思路

在请求进入缓存查询之前，先校验参数是否合法。明显非法的请求直接拒绝，不允许继续访问 Redis 和数据库。

例如：

- 用户 ID 必须是正整数。
- 商品 ID 不能超过业务允许的最大范围。
- 手机号、邮箱、订单号必须符合固定格式。
- 分页参数不能为负数，页大小不能超过上限。

```text
请求 user:-1
    |
参数校验失败
    |
直接返回错误，不查询 Redis，不查询 MySQL
```

### 5.2 优点

参数校验成本低，收益高，是防止缓存穿透的第一道防线。它能拦截大量明显错误的请求，也能减少数据库、缓存和业务代码的异常分支。

例如 `user:-1`、`user:abc` 这类请求根本不需要进入缓存层，更不应该查询数据库。

### 5.3 缺点

参数校验只能拦截格式非法或超出范围的请求。对于格式合法但业务上不存在的数据，例如 `user:100000000`，参数校验无法判断它是否真的存在。

因此，参数校验通常不能单独解决缓存穿透，需要和空值缓存、布隆过滤器一起使用。

### 5.4 使用建议

参数校验应尽量靠近请求入口，例如 Controller、网关或 RPC 接口层。越早拒绝非法请求，系统消耗越小。

不要只在数据库查询前校验，因为那时请求已经穿过了缓存逻辑，仍然会造成不必要的处理开销。

## 6. 解决方案四：接口限流和熔断

### 6.1 基本思路

当某个接口、用户、IP 或 key 的请求频率异常时，系统可以通过限流减少进入后端的请求数量。如果数据库已经出现压力，还可以通过熔断快速失败，保护核心服务。

限流常见维度包括：

- 按用户限流。
- 按 IP 限流。
- 按接口限流。
- 按 key 或业务 ID 限流。

例如同一个 IP 在 1 秒内请求了大量不存在的商品 ID，可以直接限制该 IP 的访问频率。

### 6.2 优点

限流和熔断能在流量异常时保护系统，避免数据库被突发请求压垮。它们不只用于缓存穿透，也适合防刷、防爬虫、防接口滥用等场景。

当攻击者不断构造随机 key 时，空值缓存可能无法有效命中，布隆过滤器也可能仍有少量误判请求进入后端。此时限流可以限制总体请求量。

### 6.3 缺点

限流不会判断数据是否存在，它只是控制访问频率。如果规则设置过严，可能误伤正常用户；如果设置过松，又无法有效保护系统。

熔断会牺牲一部分可用性。触发熔断后，即使部分请求本来可以成功，也可能被快速失败。

### 6.4 使用建议

限流适合放在网关层、应用入口层或核心查询接口前。规则应该结合业务访问特征设置，并配合监控动态调整。

对于读多写少的缓存查询接口，可以考虑：

- 普通用户按用户 ID 限流。
- 未登录请求按 IP 限流。
- 热点或异常 key 单独限流。
- 数据库错误率升高时触发熔断或降级。

## 7. 方案对比

| 方案 | 主要作用 | 优点 | 缺点 | 适合场景 |
| --- | --- | --- | --- | --- |
| 缓存空值 | 缓存不存在结果 | 简单直接，容易落地 | 占用额外内存，可能短暂不一致 | 重复访问同一个不存在 key |
| 布隆过滤器 | 提前判断 key 是否可能存在 | 内存小，适合拦截大量随机 key | 有误判，不擅长删除 | 数据量大、非法 key 多 |
| 参数校验 | 拒绝明显非法请求 | 成本低，入口处即可拦截 | 不能识别业务上不存在的数据 | 所有对外接口 |
| 限流和熔断 | 控制异常流量 | 能保护系统整体稳定性 | 可能误伤正常请求 | 攻击、防刷、突发流量 |

## 8. 实际开发中的组合方案

实际项目中通常不会只使用一种方案，而是多种方案组合：

```text
请求进入
    |
参数校验
    |
限流
    |
布隆过滤器判断
    |
查询 Redis
    |
未命中查询数据库
    |
数据库不存在则缓存空值
```

一种常见组合是：

1. 在接口入口做参数校验，拦截明显非法请求。
2. 对用户、IP、接口做限流，防止异常流量打穿系统。
3. 对商品 ID、用户 ID 这类核心数据使用布隆过滤器，拦截大量不存在 key。
4. 数据库查不到结果时缓存空值，并设置较短过期时间。

这样可以同时处理几类问题：

- 参数明显错误：由参数校验拦截。
- 大量随机不存在 key：由布隆过滤器拦截。
- 重复访问不存在 key：由空值缓存拦截。
- 恶意高频请求：由限流和熔断控制。

## 9. 总结

缓存穿透的本质是请求的数据不存在，导致缓存无法发挥拦截作用，请求持续进入数据库。

解决缓存穿透时，应优先做到：

1. 非法参数不要进入缓存和数据库查询流程。
2. 不存在的数据要有短期缓存，避免重复查询数据库。
3. 大规模 key 存在性判断可以使用布隆过滤器。
4. 异常高频请求要通过限流、熔断等手段保护系统。

在简单业务中，参数校验加缓存空值通常已经足够。在高并发、大流量、容易被扫描的业务中，建议再引入布隆过滤器和限流策略。
