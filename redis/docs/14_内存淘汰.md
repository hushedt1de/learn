# Redis 内存淘汰

## 1. 什么是内存淘汰

Redis 的数据主要保存在内存中。内存读写速度很快，但内存容量是有限的。如果持续写入数据，而没有删除、过期或淘汰机制，Redis 使用的内存会不断增长，最终可能导致服务器内存不足。

内存淘汰是指：**当 Redis 使用的内存达到配置上限时，Redis 按照一定策略删除部分 key，释放内存给新的写入请求使用**。

简单流程如下：

```text
客户端写入新数据
    |
Redis 检查内存是否超过 maxmemory
    |
如果没有超过：正常写入
    |
如果超过：按淘汰策略删除部分 key
    |
释放出内存后继续写入
```

Redis 通过 `maxmemory` 配置内存上限：

```conf
maxmemory 2gb
```

如果没有配置 `maxmemory`，Redis 会尽量使用系统可用内存。生产环境通常不建议完全不限制内存，因为 Redis 占用过多内存可能导致操作系统频繁交换内存，甚至触发 OOM，直接杀掉 Redis 进程。

## 2. 为什么需要内存淘汰

### 2.1 防止 Redis 占满机器内存

Redis 是内存数据库，所有 key、value、过期时间、对象元数据、复制缓冲区等都会占用内存。

如果没有上限，业务流量异常、缓存 key 膨胀、代码 bug、攻击请求等都可能让 Redis 内存不断增长。

```text
持续写入缓存
    |
Redis 内存越来越大
    |
机器内存不足
    |
Redis 变慢、被系统杀掉或影响同机其他服务
```

配置 `maxmemory` 和合理的淘汰策略，可以让 Redis 在接近内存上限时主动释放部分数据，避免把风险扩大到操作系统层面。

### 2.2 让缓存系统保持可用

很多 Redis 实例主要用作缓存。缓存数据通常可以从数据库、接口或计算逻辑重新生成，不一定需要永久保存。

当内存不足时，与其让 Redis 拒绝写入或进程崩溃，不如主动淘汰一部分不重要、访问频率低或已经接近过期的数据。

例如商品详情缓存：

```text
热门商品缓存：经常访问，应该尽量保留
冷门商品缓存：很少访问，可以优先淘汰
```

内存淘汰可以让有限内存更多留给高价值数据，提高缓存命中率。

### 2.3 应对缓存数据规模不可控

有些业务无法准确预估缓存规模。例如：

- 搜索结果缓存。
- 用户维度缓存。
- 接口响应缓存。
- 临时活动缓存。
- 推荐列表缓存。

如果每个请求都可能产生新 key，缓存数据量可能远超预期。即使每个 key 都设置了过期时间，也可能在过期前堆积到很大。

内存淘汰相当于给缓存层加了一道容量保护：即使 key 数量增长过快，Redis 也会在达到上限后按策略清理部分数据。

### 2.4 过期删除不能替代内存淘汰

Redis 支持给 key 设置过期时间：

```redis
SET product:1001 "{...}" EX 3600
```

但过期删除和内存淘汰解决的问题不同：

- 过期删除：key 到期后删除，关注数据生命周期。
- 内存淘汰：内存达到上限后删除，关注内存容量。

即使所有缓存 key 都设置了过期时间，也可能在同一时间段内占用大量内存。只依赖过期时间，不能保证 Redis 内存始终在安全范围内。

## 3. Redis 什么时候触发内存淘汰

Redis 在执行写命令时，会检查当前内存是否超过 `maxmemory`。如果超过，就根据 `maxmemory-policy` 配置的策略淘汰 key。

常见写命令包括：

- `SET`
- `HSET`
- `LPUSH`
- `SADD`
- `ZADD`
- `INCR`
- `EXPIRE`

如果淘汰后释放了足够内存，写命令继续执行。如果无法释放足够内存，Redis 会根据策略返回错误。

典型错误如下：

```text
OOM command not allowed when used memory > 'maxmemory'
```

需要注意，内存淘汰通常发生在写入阶段。读命令一般不会主动触发大规模淘汰。

## 4. 常见内存淘汰策略

Redis 通过 `maxmemory-policy` 配置内存淘汰策略：

```conf
maxmemory-policy allkeys-lru
```

常见策略可以分为三类：

1. 不淘汰，内存不足时拒绝写入。
2. 只在设置了过期时间的 key 中淘汰。
3. 在所有 key 中淘汰。

不同策略适合不同业务，不能只看名字选择。

## 5. noeviction：不淘汰

### 5.1 基本思路

`noeviction` 表示 Redis 不主动淘汰 key。当内存超过 `maxmemory` 后，如果继续执行可能增加内存的写命令，Redis 会直接返回错误。

```conf
maxmemory-policy noeviction
```

流程如下：

```text
写入新数据
    |
内存超过 maxmemory
    |
不删除任何 key
    |
写命令返回 OOM 错误
```

读命令通常仍然可以执行。

### 5.2 适合场景

`noeviction` 适合 Redis 中的数据不能被随便删除的场景。

例如：

- Redis 作为辅助存储保存重要状态。
- Redis 保存队列、Stream 消息或任务进度。
- 业务不能接受 key 被自动删除。

这类场景下，与其让 Redis 自动删数据，不如让写入失败并触发报警，由业务或运维介入处理。

### 5.3 缺点

`noeviction` 会牺牲写入可用性。一旦内存达到上限，写请求会失败，可能影响业务。

如果 Redis 主要用于缓存，通常不适合使用 `noeviction`，因为缓存本来就是可以丢弃和重建的。

## 6. volatile-lru：在过期 key 中淘汰最近最少使用的 key

### 6.1 基本思路

`volatile-lru` 只从设置了过期时间的 key 中选择淘汰对象，并优先淘汰最近最少使用的 key。

```conf
maxmemory-policy volatile-lru
```

`volatile` 表示只看有过期时间的 key；`lru` 表示 Least Recently Used，最近最少使用。

例如 Redis 中有这些 key：

```text
user:1001        有过期时间，最近访问过
product:1001     有过期时间，很久没访问
config:global    没有过期时间
```

当内存不足时，`volatile-lru` 只会在 `user:1001` 和 `product:1001` 中选择淘汰，不会淘汰 `config:global`。

### 6.2 适合场景

适合 Redis 中同时保存缓存数据和少量不希望被淘汰的数据，并且缓存 key 都设置了过期时间。

例如：

- 商品详情、用户信息等缓存设置过期时间。
- 系统配置、计数器等 key 不设置过期时间，希望尽量保留。

### 6.3 缺点

如果设置了过期时间的 key 很少，Redis 可淘汰的对象就很少。内存不足时，即使还有很多没有过期时间的 key，Redis 也不会淘汰它们，最终可能返回 OOM 错误。

因此使用 `volatile-*` 策略时，要确保可淘汰的缓存 key 都设置了过期时间。

## 7. allkeys-lru：在所有 key 中淘汰最近最少使用的 key

### 7.1 基本思路

`allkeys-lru` 会在所有 key 中选择最近最少使用的 key 进行淘汰，不管 key 有没有设置过期时间。

```conf
maxmemory-policy allkeys-lru
```

`allkeys` 表示所有 key 都可能被淘汰；`lru` 表示优先淘汰最近最少使用的数据。

例如：

```text
product:hot      最近频繁访问
product:cold     很久没访问
config:global    偶尔访问
```

当内存不足时，Redis 会倾向于淘汰长时间没有访问的 key。

### 7.2 适合场景

`allkeys-lru` 是缓存场景中很常用的策略。它适合：

- Redis 主要作为缓存使用。
- 大部分数据都可以被删除后重新加载。
- 热点数据访问明显比冷门数据频繁。

例如商品详情缓存、文章缓存、用户信息缓存、接口响应缓存，都可以考虑使用 `allkeys-lru`。

### 7.3 缺点

因为所有 key 都可能被淘汰，所以不要在同一个 Redis 实例中混放重要业务状态和普通缓存。

例如把支付状态、任务进度、缓存数据都放在同一个 Redis 实例里，并配置 `allkeys-lru`，就可能导致重要状态被淘汰。

如果必须混用，应该按业务拆分 Redis 实例，或者至少使用不同数据库和更谨慎的策略。但从隔离性看，拆分实例更清晰。

## 8. volatile-lfu：在过期 key 中淘汰访问频率最低的 key

### 8.1 基本思路

`volatile-lfu` 只从设置了过期时间的 key 中选择淘汰对象，并优先淘汰访问频率最低的 key。

```conf
maxmemory-policy volatile-lfu
```

`lfu` 表示 Least Frequently Used，最不经常使用。

LRU 关注“最近有没有被访问”，LFU 关注“一段时间内访问次数多不多”。

例如：

```text
key A：过去一小时访问 1000 次，最近 1 分钟没访问
key B：过去一小时访问 3 次，刚刚访问过一次
```

LRU 可能更倾向保留刚刚访问过的 `key B`；LFU 更倾向保留总体访问频率高的 `key A`。

### 8.2 适合场景

适合热点比较稳定的缓存场景。

例如：

- 热门商品。
- 热门文章。
- 热门榜单。
- 高频访问的基础数据。

如果某些 key 长期访问量明显高于其他 key，LFU 通常比 LRU 更容易保留真正的热点数据。

### 8.3 缺点

LFU 对访问频率建模更复杂，不是简单记录真实访问次数。Redis 使用近似计数和衰减机制来节省内存。

如果热点变化非常快，LFU 可能没有 LRU 反应灵敏。例如短时间突发热点刚出现时，访问频率还没积累起来，可能不如最近访问维度直接。

## 9. allkeys-lfu：在所有 key 中淘汰访问频率最低的 key

### 9.1 基本思路

`allkeys-lfu` 会在所有 key 中选择访问频率最低的 key 进行淘汰，不管 key 是否设置过期时间。

```conf
maxmemory-policy allkeys-lfu
```

它比 `allkeys-lru` 更关注长期访问热度，而不是最近一次访问时间。

### 9.2 适合场景

适合 Redis 主要作为缓存，并且热点数据相对稳定的业务。

例如：

- 长期热门商品详情。
- 长期热门文章。
- 公共配置和字典缓存。
- 高频访问的用户资料缓存。

如果缓存数据都可以被重建，并且希望 Redis 尽量保留高频访问数据，`allkeys-lfu` 是一个常见选择。

### 9.3 缺点

和 `allkeys-lru` 一样，`allkeys-lfu` 会淘汰所有 key，包括没有设置过期时间的 key。

因此它不适合和重要状态数据混用。只要配置了 `allkeys-*`，就应该默认认为这个 Redis 实例里的任何 key 都可能被删除。

## 10. volatile-random：在过期 key 中随机淘汰

### 10.1 基本思路

`volatile-random` 只从设置了过期时间的 key 中随机选择 key 淘汰。

```conf
maxmemory-policy volatile-random
```

它不关心 key 最近是否访问过，也不关心访问频率，只要 key 设置了过期时间，就可能被随机选中。

### 10.2 适合场景

适合数据价值差异不大、访问分布比较均匀的缓存。

例如：

- 临时验证码缓存。
- 一次性 token。
- 短生命周期临时数据。
- 对命中率要求不高的缓存。

### 10.3 缺点

随机淘汰无法优先保留热点数据。如果 Redis 中有明显热点 key，随机策略可能把热点 key 删除，导致缓存命中率下降。

多数常规缓存场景下，随机策略不如 LRU 或 LFU 稳定。

## 11. allkeys-random：在所有 key 中随机淘汰

### 11.1 基本思路

`allkeys-random` 会从所有 key 中随机选择 key 淘汰。

```conf
maxmemory-policy allkeys-random
```

它的规则非常简单：只要 Redis 内存不足，任何 key 都可能被随机删除。

### 11.2 适合场景

适合所有 key 都是可丢弃数据，并且数据之间没有明显冷热差异的场景。

例如一些短期缓存、临时计算结果缓存，或者测试环境中的缓存实例。

### 11.3 缺点

随机淘汰对缓存命中率不友好。即使某个 key 访问很频繁，也可能被随机淘汰。

生产缓存通常更推荐 `allkeys-lru` 或 `allkeys-lfu`，除非业务明确不关心热点保留。

## 12. volatile-ttl：在过期 key 中淘汰剩余时间最短的 key

### 12.1 基本思路

`volatile-ttl` 只从设置了过期时间的 key 中选择淘汰对象，并优先淘汰剩余存活时间最短的 key。

```conf
maxmemory-policy volatile-ttl
```

例如：

```text
key A：10 秒后过期
key B：5 分钟后过期
key C：1 小时后过期
```

当内存不足时，Redis 更倾向先淘汰 `key A`，因为它本来就快过期了。

### 12.2 适合场景

适合过期时间能体现数据价值的场景。

例如：

- 越接近过期越不重要的临时数据。
- 会话、验证码、短期 token。
- 活动临时缓存。

如果业务设计中 TTL 越短表示数据越不重要，`volatile-ttl` 比随机淘汰更符合直觉。

### 12.3 缺点

`volatile-ttl` 不关心访问热度。即使某个 key 很热门，只要它剩余 TTL 很短，也可能被优先淘汰。

另外，它只会淘汰设置了过期时间的 key。如果可淘汰 key 不足，仍然可能出现 OOM。

## 13. LRU、LFU、TTL 和 Random 的区别

不同策略的关注点不同：

| 策略类型 | 关注点 | 优先淘汰 | 适合场景 |
| --- | --- | --- | --- |
| LRU | 最近访问时间 | 最近最少使用的 key | 热点随时间变化的缓存 |
| LFU | 访问频率 | 访问次数较少的 key | 热点相对稳定的缓存 |
| TTL | 剩余过期时间 | 快过期的 key | TTL 能体现数据价值的缓存 |
| Random | 随机 | 随机 key | 数据价值差异不大的缓存 |

简单理解：

- 希望保留最近访问的数据，选 LRU。
- 希望保留长期高频访问的数据，选 LFU。
- 希望优先删快过期的数据，选 TTL。
- 数据都差不多，且不关心命中率，可以选 Random。

## 14. allkeys 和 volatile 的区别

`allkeys-*` 和 `volatile-*` 的区别在于淘汰范围。

| 策略范围 | 淘汰对象 | 特点 |
| --- | --- | --- |
| allkeys | 所有 key | 内存压力下释放空间更充分，但任何 key 都可能被删 |
| volatile | 只有设置了过期时间的 key | 可以保护没有过期时间的 key，但可淘汰对象可能不足 |

如果 Redis 只做缓存，并且所有数据都可以被重建，通常优先考虑 `allkeys-lru` 或 `allkeys-lfu`。

如果 Redis 中混有一些不能被淘汰的数据，可以考虑 `volatile-lru`、`volatile-lfu` 或 `volatile-ttl`，并确保普通缓存 key 都设置过期时间。

更稳妥的做法是按数据重要性拆分 Redis 实例：

- 缓存实例：使用 `allkeys-lru` 或 `allkeys-lfu`。
- 状态实例：使用 `noeviction`，并配合监控和扩容。

## 15. 怎么选择内存淘汰策略

### 15.1 纯缓存场景

如果 Redis 主要作为缓存，数据丢失后可以从数据库或其他来源重新加载，推荐：

```conf
maxmemory-policy allkeys-lru
```

或者在热点比较稳定时使用：

```conf
maxmemory-policy allkeys-lfu
```

这类策略可以让 Redis 在内存不足时主动释放冷数据，尽量保留热点数据。

### 15.2 缓存和重要数据混用

如果同一个 Redis 实例里既有缓存，又有不希望被淘汰的重要数据，可以考虑：

```conf
maxmemory-policy volatile-lru
```

前提是缓存 key 都设置了过期时间，重要 key 不设置过期时间。

但这种混用方式仍然有风险。因为一旦缓存 key 不足以释放内存，Redis 仍可能拒绝写入。更清晰的方案是把缓存和重要数据拆到不同 Redis 实例。

### 15.3 重要状态数据

如果 Redis 中保存的是不能随便丢的数据，例如任务状态、消息、计数、分布式协调状态等，推荐：

```conf
maxmemory-policy noeviction
```

这样 Redis 不会自动删除 key。内存不足时写入失败，业务可以感知错误，监控系统也可以及时报警。

### 15.4 临时数据

如果 Redis 保存的是短生命周期临时数据，例如验证码、短期 token、临时标记，可以考虑：

```conf
maxmemory-policy volatile-ttl
```

或者：

```conf
maxmemory-policy volatile-random
```

这类数据通常都设置了过期时间，且单个 key 的价值不高。

## 16. 使用内存淘汰的注意事项

### 16.1 不要把淘汰当成正常删除机制

内存淘汰是 Redis 内存不足时的保护机制，不应该替代业务上的过期时间和删除逻辑。

缓存 key 仍然应该设置合理 TTL，业务删除或更新数据时也应该主动更新或删除相关缓存。

### 16.2 不要混放不同重要性的 key

如果一个 Redis 实例中同时保存普通缓存和重要业务状态，很难选择合适的淘汰策略。

使用 `allkeys-*` 可能删除重要状态；使用 `volatile-*` 又可能因为可淘汰 key 不足导致写入失败。

更好的做法是按用途拆分实例：

- 缓存数据一个 Redis。
- 分布式锁一个 Redis 或独立逻辑库。
- Stream、队列、任务状态单独 Redis。

### 16.3 监控内存和淘汰指标

生产环境要关注 Redis 的内存使用和淘汰情况。

常见指标包括：

- `used_memory`：Redis 当前使用内存。
- `maxmemory`：配置的内存上限。
- `evicted_keys`：被淘汰的 key 数量。
- `expired_keys`：过期删除的 key 数量。
- `keyspace_hits`：缓存命中次数。
- `keyspace_misses`：缓存未命中次数。

可以通过 `INFO memory` 和 `INFO stats` 查看：

```redis
INFO memory
INFO stats
```

如果 `evicted_keys` 持续快速增长，说明 Redis 长期处于内存压力下。需要评估是否要增加内存、优化 key 大小、缩短 TTL、拆分实例或调整淘汰策略。

### 16.4 关注大 key

内存淘汰是按 key 淘汰，而不是按字段、元素或部分内容淘汰。

如果一个 hash、list、set 或 zset 特别大，淘汰它会一次性删除大量数据；如果它很热，又可能长期占用大量内存。

因此要避免大 key：

- 大 hash 可以按业务维度拆分。
- 大 list 可以控制长度。
- 大 set 和 zset 可以分片或定期清理。
- 大 value 可以压缩或拆成多个小 key。

## 17. 小结

内存淘汰是 Redis 在达到 `maxmemory` 后，为了释放内存而主动删除 key 的机制。它主要用于保护 Redis 实例，避免内存无限增长导致服务不可用。

常见策略可以这样记：

- `noeviction`：不淘汰，内存满了拒绝写入。
- `allkeys-lru`：所有 key 中淘汰最近最少使用的 key。
- `allkeys-lfu`：所有 key 中淘汰访问频率最低的 key。
- `volatile-lru`：只在有过期时间的 key 中按 LRU 淘汰。
- `volatile-lfu`：只在有过期时间的 key 中按 LFU 淘汰。
- `volatile-ttl`：只在有过期时间的 key 中优先淘汰快过期的 key。
- `volatile-random`：只在有过期时间的 key 中随机淘汰。
- `allkeys-random`：所有 key 中随机淘汰。

如果 Redis 主要做缓存，通常优先考虑 `allkeys-lru` 或 `allkeys-lfu`。如果 Redis 保存重要状态，应该谨慎使用淘汰策略，通常更适合 `noeviction` 并配合监控、扩容和业务降级。
