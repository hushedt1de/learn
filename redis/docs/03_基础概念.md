# Redis 基础概念

## 1. Redis 是什么

Redis（Remote Dictionary Server）是一种开源的、基于内存的键值型数据存储系统。它以 `key-value` 的形式保存数据，并提供字符串（String）、哈希（Hash）、列表（List）、集合（Set）、有序集合（Sorted Set）、流（Stream）等多种数据结构。

Redis 的主要特点包括：

- **读写速度快**：数据主要存放在内存中，适合低延迟、高并发的访问场景。
- **数据结构丰富**：不仅能保存简单字符串，还能直接操作列表、集合、有序集合等结构。
- **支持数据持久化**：可以通过 RDB 快照或 AOF 日志将内存中的数据保存到磁盘，在服务重启后恢复。
- **支持过期时间**：可以为键设置生存时间，到期后由 Redis 自动删除。
- **支持高可用和扩展**：可通过主从复制、哨兵（Sentinel）和集群（Cluster）实现容灾及水平扩展。

Redis 通常作为缓存或辅助存储使用。在特定业务中也可以作为主要数据库，但需要根据数据可靠性、查询方式和容量等要求谨慎设计。

## 2. Redis 有什么作用

Redis 的核心作用是利用内存的高速读写能力，减少应用访问后端数据库或其他慢速服务的次数，从而提升系统性能和并发处理能力。

常见作用如下：

1. **缓存热点数据**  
   将频繁读取的数据保存在 Redis 中，避免每次都查询 MySQL，从而降低数据库压力并缩短响应时间。

2. **保存临时数据**  
   Redis 支持为键设置过期时间，适合保存验证码、登录状态、临时令牌等有明确有效期的数据。

3. **提供原子操作**  
   Redis 的命令通常具有原子性，可以安全地执行计数、库存扣减等操作。但复杂业务仍需使用 Lua 脚本或事务保证多个操作的整体原子性。

4. **实现消息传递**  
   Redis 提供发布订阅（Pub/Sub）和 Stream。Pub/Sub 适合实时通知但不持久化消息；Stream 支持消息持久化、消费组和消息确认，更适合可靠的消息处理。

5. **支持分布式协作**  
   可以利用键的过期时间及条件写入等能力实现分布式锁、限流器等组件。实现分布式锁时需要正确处理唯一标识、超时和原子释放等问题。

## 3. 命令执行与原子性

Redis 服务端按事件循环处理命令。单条命令在执行期间不会与另一条普通命令交错，因此 Redis 单命令操作具有原子性。

原子性只覆盖单条命令，不覆盖由多条命令组成的业务流程。例如下面这种流程并不安全：

```text
GET article:1001:views
应用层加一
SET article:1001:views 新值
```

客户端依次发送 `GET` 和 `SET`，两条命令之间可能插入其他客户端的操作。

并发请求可能读取到相同旧值并互相覆盖，因此计数应优先使用 `INCR`、`HINCRBY` 这类单命令增量操作。

需要把多步读写作为一个不可分割操作时，通常使用 Lua 脚本或 Redis Functions；也可以根据场景使用 `WATCH`、`MULTI`、`EXEC` 实现乐观锁。

## 4. Redis 的主要应用场景

### 4.1 数据缓存

缓存商品详情、用户信息、配置项等高频访问数据。常见流程是先查询 Redis；如果未命中，再查询 MySQL，并将结果写入 Redis。

使用缓存时需要考虑缓存穿透、缓存击穿、缓存雪崩，以及缓存与数据库之间的数据一致性。

### 4.2 Session 和登录状态

在分布式系统中，多台应用服务器可以将 Session、访问令牌或用户登录状态统一保存到 Redis，避免状态只存在于某一台服务器中。

### 4.3 验证码和临时令牌

验证码、密码重置令牌、接口幂等令牌等数据只需短期保存。Redis 的键过期机制可以在有效期结束后自动清理这些数据。

### 4.4 计数器

利用 `INCR`、`DECR` 等原子命令实现文章阅读量、点赞数、接口调用次数等实时计数。

### 4.5 排行榜

使用有序集合（Sorted Set）保存成员及其分数，可以高效地完成按分数排序、获取排名和查询指定区间等操作，适用于积分榜、热度榜等场景。

### 4.6 限流

通过计数器、有序集合或 Lua 脚本实现固定窗口、滑动窗口、令牌桶等限流策略，用于限制用户或接口在一定时间内的请求次数。

### 4.7 消息队列和实时通知

简单实时通知可以使用 Pub/Sub；需要消息留存、消费组及失败重试能力时可以使用 Stream。对于复杂、强可靠的消息系统，通常还需要评估 Kafka、RabbitMQ 等专用消息中间件。

### 4.8 分布式锁

多个服务实例需要互斥访问共享资源时，可以通过 Redis 实现分布式锁。例如避免同一订单被重复处理。分布式锁并非简单的“设置一个键”，还要处理锁超时、误释放、续期及故障场景。

## 5. Redis 和 MySQL 的区别

Redis 和 MySQL 的定位不同。Redis 更擅长高速访问和特定数据结构操作，MySQL 更擅长可靠地保存结构化业务数据及执行复杂查询。

| 对比项 | Redis | MySQL |
| --- | --- | --- |
| 数据库类型 | 键值型 NoSQL 数据库 | 关系型数据库 |
| 主要存储位置 | 内存，支持持久化到磁盘 | 磁盘，利用内存进行缓存 |
| 数据模型 | Key-Value 及多种数据结构 | 表、行、列及表之间的关系 |
| 查询方式 | 主要通过键和数据结构命令访问 | 使用 SQL，支持条件、关联、聚合等复杂查询 |
| 性能特点 | 内存读写，通常延迟更低、吞吐量更高 | 相对较慢，但查询能力和事务能力更完整 |
| 事务 | 提供事务命令和 Lua 脚本，但语义与关系型数据库事务不同 | 支持较完整的 ACID 事务（具体能力与存储引擎有关） |
| 数据容量 | 主要受可用内存限制，内存成本较高 | 主要受磁盘容量限制，适合存储大量数据 |
| 数据可靠性 | 支持 RDB、AOF、复制等机制，但配置不当或故障时可能丢失最近的数据 | 通常更适合持久保存核心业务数据 |
| 典型用途 | 缓存、排行榜、计数器、Session、限流、临时数据 | 用户、订单、商品等需要长期保存和复杂查询的核心数据 |

### Redis 能否替代 MySQL

多数情况下不能简单替代。常见架构是将两者配合使用：

- MySQL 作为主要数据库，保存需要长期持久化的核心业务数据。
- Redis 作为缓存或辅助存储，保存热点数据、临时状态及适合特定数据结构处理的数据。

例如查询商品信息时，应用先读取 Redis；缓存未命中时再查询 MySQL，并将查询结果写回 Redis。这种方式兼顾了 MySQL 的数据管理能力和 Redis 的访问速度。
