# Redis String 实现原理

## Redis 对象与内部编码

Redis 中每个键对应的值，并不是直接裸露地保存为字符串、列表或哈希表，而是由一个统一的 Redis 对象描述。

这个对象会记录几类重要信息：

- **逻辑类型**：值对外表现出来的 Redis 类型，例如 `string`、`list`、`hash`。
- **内部编码**：Redis 为这个值选择的底层存储方式。对于 String，常见编码包括 `int`、`embstr` 和 `raw`。
- **数据指针**：指向真正保存数据的底层结构。对于 String，可能直接保存整数值，也可能指向 SDS 字符串。

因此，String 是用户看到的逻辑类型，而 `int`、`embstr`、`raw` 是 Redis 在内部为了节省内存或提高效率选择的不同实现方式。

String 会根据值的内容和长度选择紧凑的内部编码，常见情况包括：

- **整数编码**：能够表示为整数的值可以直接以整数形式保存在对象中，减少额外分配，并且不需要解析字符串。
- **紧凑字符串编码（embstr）**：较短的字符串通常将对象元数据和字符串数据放在一次连续内存分配中，减少分配次数并改善缓存局部性。
- **普通字符串编码（raw）**：较长或被修改后的字符串通常使用对象与字符串数据分离的形式，便于扩容和修改。

可使用 `OBJECT ENCODING key` 观察当前内部编码：

```redis
SET demo:number 123
OBJECT ENCODING demo:number
```

## SDS：简单动态字符串

Redis 的字符串数据主要由 SDS（Simple Dynamic String）表示，而不是直接使用以 `\0` 结尾的 C 字符串。SDS 的结构在概念上包含：

- 当前已使用的字节长度；
- 当前可用或已分配空间；
- 标识头部布局的类型信息；
- 保存实际字节数据的缓冲区。

这带来几个重要特性：

1. **二进制安全**：长度单独记录，内容可以包含 `\0` 和任意二进制字节。
2. **获取长度高效**：不需要从头扫描字符串，读取已记录的长度即可。
3. **降低扩容成本**：修改字符串时可以根据需要扩展空间，避免每次追加一个字节都重新分配。
4. **兼容部分 C 字符串函数**：缓冲区末尾仍保留终止符，但终止符不计入逻辑长度。

## 时间和空间复杂度

常见操作的复杂度如下：

| 命令 | 典型时间复杂度 | 说明 |
| --- | --- | --- |
| `GET`、`SET`、`INCR` | O(1) | O(1) 不代表与值大小无关；大值仍有复制和网络传输成本 |
| `MGET`、`MSET` | O(N) | N 为键的数量 |
| `STRLEN` | O(1) | 长度保存在 SDS 元数据中 |
| `APPEND` | 均摊 O(1) | 创建新键时还与追加值长度有关 |
| `GETRANGE` | O(N) | N 为返回的数据长度 |
| `SETRANGE` | O(1) | 扩展值时与新增长度相关 |
| `BITCOUNT` | O(N) | N 为扫描的字节数 |

String 的内存占用不仅包括值本身，还包括键、Redis 对象、SDS 头、哈希表节点、内存分配器对齐和可能的碎片。因此，用大量很短的键值对保存数据时，元数据开销可能高于有效数据本身。
